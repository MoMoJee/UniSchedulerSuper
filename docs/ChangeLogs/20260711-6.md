# 2026-07-11 Planner P2-A/P2-B：可重跑导入与只读校验

## 新增能力

- 新增 `core/planner/migration.py`：按用户读取唯一 legacy source 行，构建无副作用计划；`--apply` 时才在单一数据库事务与 reversion revision 中建立 normalized 旁路投影。
- 新增 `migrate_planner_legacy` 命令。默认和 `--dry-run` 都不写数据库，`--apply` 是唯一写入开关；同一 source checksum 重复 apply 会安全跳过。
- 普通 event/todo/reminder、日程组、标签、提醒关系、待办依赖、分享关系、提前提醒与 legacy ID mapping 可导入；未知 JSON 字段保存到对象 metadata。
- 单规则段 event/reminder recurrence 导入为 master + series + EXDATE + sparse reminder state，正常预生成实例仅写 `PlannerLegacyIdMap`，不会再物化为新 event/reminder 行。
- 新增 `PlannerMigrationIssue` 隔离逻辑：非法 RRule、多个规则段、重复/缺失 legacy ID、时间/引用异常、目标冲突和精度损失都会阻止 cohort，而不是用默认值覆盖。
- 新增 `verify_planner_migration` 与 `verify_recurrence_parity`。两者均只读，比较 source checksum、实体字段、legacy ID map 与指定窗口的 occurrence 集；`--strict` 有差异时失败。
- 新增 `core.0010_planner_cohort_assignment`、`PlannerRolloutPolicy` 与 `set_planner_cohort`。用户需同时拥有已验证的所有 source state、零未解决 issue 和明确入口登记；否则即便全局配置为 normalized，策略也回退 legacy。

## 生产库只读演练

执行了 `migrate_planner_legacy --dry-run`，未传 `--apply`：39 位用户中 34 位通过当前预检，5 位被隔离。隔离原因为历史 RRule 不可规范化或 legacy item ID 缺失。没有写入 normalized 业务行、`PlannerMigrationState` 或 `PlannerMigrationIssue`，也没有切换流量。

`core.0010_planner_cohort_assignment` 已应用；该 migration 仅创建 cohort assignment 表，不导入或改写任何 Planner 业务数据。

## 验证

- 新增迁移命令测试，覆盖 dry-run 零写入、非法数据预先隔离、普通实体/关系/unknown field、幂等 apply、单规则段 series→master 映射、`verify_planner_migration` 与 `verify_recurrence_parity`。
- 本阶段仍保持 legacy JSON 是唯一业务事实源；P3 之前禁止线上读写切到 normalized 表。

## 后续

1. 分类和人工修复 dry-run 报告中的 5 位隔离用户。
2. 在一致性备份副本上执行 `--apply`，随后运行两个 `--strict` 校验命令。
3. 在通过副本 parity 的用户上执行 `--strict --mark-verified`，再显式登记 shadow cohort；未通过 parity 的用户永远不进入 cohort。
