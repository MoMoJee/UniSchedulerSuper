## DeepSeek KV Cache 实测结论

使用底层 OpenAI SDK 直接请求 DeepSeek，绕过 LangGraph / LangChain / 主 Agent 代码，验证本次命中率提升的主因。

| 测试场景 | 预热后平均命中率 | 结论 |
|---|---:|---|
| `human_prefix + api_tools + no_user` | 98.33% | 不带 user_id 也可高命中 |
| `human_prefix + api_tools + extra_body.user_id` | 98.23% | user_id 不降低命中，但不是主因 |
| `none_timestamp + api_tools + extra_body.user_id` | 98.88% | 无动态时间戳时最高 |
| `system_before_user + api_tools + extra_body.user_id` | 6.81% | runtime `SystemMessage` 会严重破坏缓存 |
| `system_front + api_tools + extra_body.user_id` | 0.00% | 动态 system 放最前完全失效 |
| `human_prefix + api_tools + tool_loop` | 96.94% | 多工具调用历史仍可高命中 |

进一步定位：

| 测试场景 | 无 tools | 47 tools | 结论 |
|---|---:|---:|---|
| `human_prefix_dynamic` | 约 94.4% | 约 97.8% | 动态内容在 user 前缀中安全 |
| `assistant_context_dynamic` | 约 95.0% | 约 98.0% | 动态内容在 assistant 区域中也不破坏 tool 前缀 |
| `interleaved_system_dynamic` | 约 90.8% | 约 7.5% | 开启 tools 后，中途 system 会破坏 DeepSeek 前置缓存单元 |
| `interleaved_system_static` | 约 90.8% | 约 7.5% | 结构本身就危险，不只是时间戳文本变化 |
| `system_front_dynamic_extra` | 0% | 0% | 最前动态 system 必然无缓存 |

结论：DeepSeek 在 tool-enabled 请求中并不是按调用方 `messages` 数组做简单最长公共前缀缓存，而是会把 `tools` 与 `system` 区域序列化成特殊前置 prompt。后续新增或变化的 `SystemMessage` 会污染该前置区域；动态 runtime context 必须放在 user message 边上。

## 后续维护规则

1. 主 Agent 开启 tools 时，禁止把秒级时间、当前轮附件、临时运行状态作为新的 `SystemMessage` 插入历史中间或最新用户消息前。
2. 动态上下文统一写入 `HumanMessage.additional_kwargs`，只在出站 materializer 中前缀到 user 内容。
3. `extra_body.user_id` 保留并按功能 suffix 隔离，但不要把它当作缓存命中率的主要手段。
4. 工具 schema 继续通过 API `tools` 字段发送，不要为了缓存改成 system prompt 文本。
5. 压缩、换 provider、切换多模态能力视为新的缓存纪元，可以接受下一轮重新建立缓存。
