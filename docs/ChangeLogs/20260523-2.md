## 配置迁移

- `config/api_keys.example.json` 已迁移到 `provider_styles + style + 三段价格`。
- 真实 `config/api_keys.json` 已做结构迁移，但仍作为敏感配置文件处理，不在 changelog 中记录密钥或完整内容。
- system 模型必须显式配置 `cost_per_1k_input_cache_hit`；未命中输入价格使用 `cost_per_1k_input_cache_miss`，旧 `cost_per_1k_input` 仅作为兼容回退。
- 本次仅处理 system 模型，用户自定义模型仍走旧兼容路径。

## 行为变化

- `update_token_usage()` 新签名优先接收 `usage_info` dict；旧签名仍兼容，但会记录 WARNING，且 cache hit 按 0 处理。
- `/api/agent/token-usage/` 保留前端现有字段，并新增 `request_record_count` 与 `last_request_record` 调试摘要。
- `reset_token_stats(current)` 继续保留 `monthly_used`，仅清空当前月模型累计。
- `AgentUsageRecord` 默认完整保留，不复制请求明细到 `UserData.agent_token_usage`。

## 经验与后续规则

1. Provider 兼容差异应进入集中 style，而不是继续散落在模型配置和调用分支里。
2. usage 抽取、计费、快照、请求明细必须共享同一个归一化结构，避免同一响应在不同调用点被重复解析。
3. 涉及真实 `api_keys.json` 的迁移只能做脱敏状态输出，不展开 diff，不输出 key/token/bearer/request body。
4. 请求级用量明细适合 ORM 表；`UserData.agent_token_usage` 只保存月度累计和历史摘要。
5. 兼容旧字段时必须 WARNING，方便后续定位还未迁移的调用点或配置。
