# 2026-07-11 核心日程正规化与 RRule 升级：P0 完成、P1 基础落地

## 变更摘要

- 阅读并按《核心日程正规化与 RRule 引擎升级方案》启动分阶段实施；当前保持 legacy JSON 为唯一业务事实源，未切换 Web、Agent、MCP、Feed 或 CalDAV 流量。
- 使用 SQLite online backup API 创建升级前一致性备份：`backup/20260711/planner_pre_upgrade_20260711_035600.sqlite3`。
- 新增 `audit_planner_legacy` 命令，审计 legacy Planner 的源行、checksum、重复 `(user,key)`、JSON 格式错误和数据规模。
- 初始审计发现 user 44 的 `reminders` 有两条完全相同的空数组源行；新增 `resolve_planner_legacy_duplicates` dry-run/`--apply` 命令后，仅删除重复副本 520、保留 519。
- 复审计结果：37 位用户、157 条 Planner 源行、0 个重复 key、0 个非法 JSON。
- 新增并应用 `core.0009_planner_normalized_foundation`：创建 EventGroup、CalendarEvent、Todo、Reminder、tags/links/share links、event/reminder recurrence、migration state/issue、ChangeSet、collection version/change 等正规化表；未导入正常业务对象。
- 新增 `PlannerTimeCodec`：统一 DATE、TZID、UTC、legacy ISO、RECURRENCE-ID 与受支持 RFC 5545 RRULE 的规范化校验。
- 新增无副作用 `RecurrenceExpander`：按窗口合并 RRULE、RDATE、EXDATE、modified/cancelled override，且不物化普通 occurrence。
- 新增 `LegacyPlannerRepository`：直接保留原始 JSON 中的未知字段，避免 `DATA_SCHEMA` 重建删除 CalDAV UID 或扩展字段。
- 将新 Planner 模型注册到 Django Admin。
- Django 测试进程不再在 Agent 图导入时连接真实 MCP；MCP 配置日志不再输出可能含密钥的 URL。

## 数据影响
