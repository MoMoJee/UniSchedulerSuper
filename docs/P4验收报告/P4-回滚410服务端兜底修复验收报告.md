# P4 Agent 回滚 410 服务端兜底修复验收报告

> 验收日期：2026-07-12  
> 结论：通过。该问题已修复并纳入 P4 最终完成范围。

## 故障与根因

MoMoJee 在 `/api/agent/rollback/to-message/` 收到 410。数据库与访问日志表明，失败会话已有消息但没有 `AgentRollbackWindow`，并且此前没有 window rotate 请求。根因是初版 P4 只在新版前端切换会话时显式创建窗口；旧页面、缓存静态资源或生命周期请求丢失时，后端安全检查只能把所有消息判定为升级前历史并返回 410。

“是否发生 Agent 工具调用”与这个 410 无关：窗口是消息回滚的授权边界，Planner transaction/snapshot 只是窗口内日程副作用的恢复载体。没有工具调用的普通聊天仍应删除消息、附件/TODO/token 状态，而不应因为事务数为 0 被拒绝。

此前测试未发现，是因为回滚用例都先显式调用 rotate，覆盖了窗口存在后的 200/409/410 分支，却缺少“已有历史但窗口缺失”和“普通聊天零 Planner 事务”两种端到端状态。

## 修复实现

- `agent_service/rollback_windows.py`：新增 `ensure_active`，同会话复用已有 active window，缺失时按当前消息末尾创建；创建新会话窗口会关闭用户的其他 active window。
- `agent_service/consumers.py`：WebSocket 建连读取历史消息数后执行服务端兜底；无法可靠取得消息数时关闭连接，绝不以 floor=0 猜测。
- `agent_service/views_api.py`：history 空/非空响应都会返回服务端窗口；缺失窗口分别以 0/当前消息总数为 floor 建立。
- `agent_service/tests/test_agent_rollback_window.py`：增加窗口缺失修复、刷新幂等、切换关闭、history 末尾 floor、普通聊天零事务回滚等覆盖。

## 测试结果

| 测试 | 结果 |
|---|---|
| rollback window + cleanup strict 定向 | 15/15 通过 |
| `core.tests + agent_service.tests` 全量 | 124/124 通过，32.965s |
| MoMoJee 失败会话认证 history smoke | 200；floor=2；旧消息不可回滚；窗口 generation=1 |
| 过期/旧 floor | 410，数据库和消息零部分修改 |
| 后续并发写冲突 | 409，数据库和消息零部分修改 |
| 普通聊天、0 个 Planner transaction | 200，消息按目标位置删除 |
| Planner 工具消息 | 200，按消息倒序恢复 snapshot 并清缓存 |

## 生产存储复验

`verify_planner_rollback_storage --strict` 已通过：SQLite integrity `ok`、foreign key violation 0、Planner UserData Version 0、旧 `Before:` Agent Revision 0、两套 legacy AgentTransaction 0、orphan Revision 0。数据库中的 42 个 UserData Version 均为 MoMoJee 的 `user_interface_settings`，约 86KB，不属于日程或 Agent 回滚历史。

当前 8080 端口没有运行中的旧项目进程。下一次按项目正常方式启动服务即可加载本次修复；浏览器建议硬刷新一次以更新静态资源，但即使页面没有主动 rotate，后端兜底也会保证新消息的窗口正确建立。
