## 审计方法与交付门槛

1. 仅在 legacy `/home/` 进行浏览器真实交互：点击、键盘、滚动、拖拽、切换、刷新、弹窗开关和错误状态；不以源码推断替代视觉记录。
2. 每个状态保存原始 PNG，文件名采用 `NNN-模块-场景-状态.png`；文档按截图编号记录入口、前置条件、动作、可见结构、动效、滚动/焦点、涉及 API 与 React 复刻验收点。
3. 第一批最少 100 张覆盖截图；完整审计按模块继续扩展，直到基线中的全部可见状态、可操作控件和错误/空态都有证据。截图数量不是替代覆盖率的指标。
4. 每一个 React 修复必须引用旧版截图编号，并有相同操作的新版截图、自动化用例和功能验收结果。

### 006 细则：日程组筛选的不可变规则

- 所有条件默认均为勾选；取消一个条件只排除该分支，不能替换为 radio 或单一 `groupId`。
- 日程组示例包含“其他、BELP及传感皮肤、BoGDB、USS升级、事务、学习、家教、技能、日常、考试、课表”等；它们可同时勾选，组合结果与事件颜色映射共同生效。
- React 复刻必须持久化整个条件集合、刷新后恢复，并在 Calendar 内以同等层级打开；不得隐藏为单一颜色下拉框。

### 007–023 的接口和状态边界

- 所有 007–023 截图均只执行 UI 打开、筛选和未提交表单分支；没有创建、编辑、删除、导入或分享真实数据。
- Calendar 状态至少拆为：`view/date`、可组合的可见性筛选、日程组色彩、模态栈、未提交表单草稿和分享组选项；任何关闭动作只影响对应模态层。
- 新版不能把“日程组色彩”“共享目的地”“RRule 配置”放进一个扁平化的 quick form。它们是旧版独立且可滚动的工作流节点。

### 024–036 的回归重点

- Todo 的“列表/四象限”是视觉与信息架构的双视图，不可只迁移数据卡片；四象限中的拖拽变更属于后续专门操作测试。
- Reminder 规则配置与 Event 的 RRule 有公共概念但不能共享错误的组件状态。创建、编辑、筛选、日历实例和左栏摘要都必须消费同一规范化规则结果。
- 本批所有表单均未提交；对已有待办仅进入编辑/滚动/退出，没有发生删除或更新。

### 源码交叉发现：重复事件范围链存在双重确认和可达性风险

- `event-manager.js` 同时保留模板内的 `deleteRecurringEventModal`，并在 `showEventEditScopeDialog()` 动态插入另一套范围 modal；后者甚至提供 `single/all/future/from_time` 四种选项。两套实现并存是范围 UI 不稳定的直接风险。
- `executeEventEditScope(..., 'delete')` 在用户选择范围后仍调用浏览器原生 `confirm()`。此前实际审计点击删除时正是该原生确认框阻塞了旧标签页；新版不得沿用此不可样式化、不可组件测试、会阻断自动化的实现。
- 因此，后续复刻的唯一目标态应是：一个 React 范围选择 dialog（this/all/future，必要时明确的自定义起点）→ 一个应用内最终确认 dialog → V2 mutation → 局部刷新/可回滚结果；不允许同时存在 legacy modal、动态 modal、浏览器 `confirm()` 三条路径。

### 198：Agent 展开聊天模态框与发现的旧版缺陷

[展开聊天模态框](assets/legacy-ui-audit/198-legacy-agent-expanded-view.png) 由右栏的“展开聊天”按钮打开。它复用同一 Agent 会话，提供更大的消息区、输入区及“今日日程 / 创建事件 / 时间分析 / 路线规划”快捷提示入口；标题栏应有新建会话和关闭控件。

本轮只读审计发现一个需要在 React 复刻中**修复而非照搬**的旧版缺陷：展开模态框中视觉为“+”的新建会话按钮写的是 `onclick="agentChat.newSession()"`，但实际对象只实现了 `createNewSession()`。因此点击会抛出 `TypeError: agentChat.newSession is not a function`，不会发送创建会话请求，也不会改变会话 ID 或消息。审计中曾误触该错误按钮，随后核验到当前用户消息数仍为 0、会话 ID 未产生、后端未写入；真实关闭控件是同一标题栏的 `span.close`，已使用它关闭弹窗。

- 展开/收起仅改变展示容器，不丢失草稿、附件、会话或上下文；
- 四个快捷提示只能填入草稿或按明确产品定义处理，不能静默发消息；
- “新建会话”需走统一的确认、回滚窗口收束、持久化创建和错误反馈流程；
- 关闭控件应具备明确可访问名称、唯一选择器和键盘焦点，避免与新建动作视觉混淆。

### 三栏分隔条：源码与真实 DOM 交叉审计

- 标准 1366 × 900 桌面页实际有两个 `div.resize-handle`：`left-resize` 位于左栏与中栏之间，`right-resize` 位于中栏与右栏之间；每个宽 8px。当前加载的布局内联基准为左/中/右 `flex-basis: 20% / 50% / 30%`。截图 193 已保留该三栏的可见基线。
- 交互脚本 `panel-resizer.js` 同时监听鼠标和触摸：左分隔条允许左栏 15–35%、中栏 30–70%；右分隔条允许中栏 30–70%、右栏 20–50%。拖动结束会触发 Calendar `updateSize()`、Todo 象限 `adjustQuadrantLayout()`，并调用 `settingsManager.onPanelLayoutChange(...)` 保存布局。
- CSS 规范为透明 8px 拖拽带、`col-resize` 光标，内部 4 × 40px 灰色把手；hover/拖动时转为蓝色。移动端媒体规则隐藏分隔条，采用垂直重排。
- 由于停止拖动会写入当前用户的布局设置，本轮不在真实账号上拖动，也不伪造“已恢复”的写入结果。React 的隔离验收必须逐一覆盖：左右两条、鼠标/触摸/键盘替代操作、最小最大边界、取消/恢复原比例、刷新持久化、窗口缩放后的 Calendar 与 Todo 重排，以及移动端隐藏/重排；保存失败不得留在半拖动状态。

这意味着新版帮助页若保留，验收必须逐项覆盖长页滚动、返回工作台、FAQ 文本与当前 V2 能力一致性、外链的新标签安全属性、邮箱隐私决策、窄屏和键盘回到顶部；不得只留下一个“帮助”标题按钮。204/205 的补拍仍受 CDP 截图超时影响，未创建虚假图片引用。
