# 前端界面重构前功能基线与技术选型调研

> 状态：审计进行中。创建日期：2026-07-13。本文是从 Django Template + 原生 JavaScript 迁移到现代前端框架前的功能冻结清单。每读完一个模板、模块或样式域立即追加记录；没有记录为“已审计”的文件不得在重构中视为可删除。

## 0. 审计范围与方法

### 0.1 范围

已发现 51 个界面相关文件：

- 核心页面模板 20 个（含首页、认证、设置、帮助、演示页和局部片段）；
- 自研 JavaScript 模块 15 个；
- 自研 CSS 5 个；
- 文件服务模板、JS、CSS 3 个；
- 本地第三方/构建产物 8 个。

本次重构的“功能不遗漏”范围包含：可见页面、弹窗、快捷操作、键盘/触摸交互、主题和布局持久化、错误/加载状态、所有 HTTP/WebSocket 调用、数据刷新策略以及附件/回滚等隐藏流程。第三方库源码不逐行复述，但记录其版本、能力和替换约束。

### 0.2 现行数据与接口约束

- Planner 只能请求 `/api/v2/`；读取 occurrence 必须有有限 `[from,to)`，写入需要 `expected_version`，重复项需要 `occurrence_ref`。
- `/get_calendar/events/`、旧 todo/reminder/event API 均已封存。审计中若发现遗留调用，将记录为**重构时必须消除**，不能视为兼容需求。
- Agent 历史、附件、回滚资格以服务端为准；WebSocket 仅用于实时事件。
- 新实现必须保留现有 Django Session + CSRF 和 Agent WebSocket 的认证边界。

## 1. 已审计记录

### 1.1 全站 CDN 与公共静态加载：`core/templates/_cdn_init.html`

- **入口与展示**：不是独立页面；由多个页面在 `<head>` include。提供 BootCDN（国内）/ jsDelivr（国际）的 Bootstrap CSS 双源加载。
- **行为与状态**：在 `localStorage` 保存 `cdn_pref`、`cdn_t`、`cdn_ver`；每小时用 HEAD 请求竞速两个 CDN，CSS 加载超过 3 秒或失败时切到备用源。
- **重构保留要求**：保留“离线/网络差时资源可用”的能力，但由 Vite 构建产物、自托管静态资源和必要的 CDN fallback 取代运行时竞速脚本。主题、业务状态不得与 CDN 偏好耦合。
- **风险**：全局 `__cdnCSS` 与手写 URL 分散在模板；不同页面 Bootstrap 版本不一致（5.1/5.3），必须在新工程统一。

### 1.2 公开落地页：`core/templates/index.html`

- **展示**：项目介绍 Hero、日程/提醒/待办/协作/Agent/iCalendar 六张特性卡、技术栈标签、非商业平台说明、GitHub/备案页脚。
- **交互**：亮/暗主题圆形按钮；仅保存 `localStorage.theme`，根据系统主题初始化。背景旋转、图标浮动、卡片 hover 都是 CSS 动画。
- **接口**：无业务 API；模板变量为 `current_year`、`beian_info`。
- **重构保留要求**：作为公开 marketing/landing route 独立于登录后应用；保留 SEO meta、备案条件显示、GitHub 外链、暗色主题与可减少动效偏好。

### 1.3 README 与帮助：`about.html`、`help.html`

- **`about.html` 展示**：将服务端提供的 `readme_content` 以 `github-markdown.css` 渲染为阅读页；没有互动或 API。
- **`help.html` 展示**：静态使用说明，列出日程 CRUD/拖拽/RRule/DDL、提醒筛选、Agent、搜索、设置入口、FAQ、邮箱与 GitHub。固定“返回主页”按钮。
- **重构保留要求**：帮助内容改为受版本控制的 MDX/Markdown 页面或知识库路由；Markdown 必须继续进行可信 HTML/安全渲染。文案中涉及旧 UI（例如“左侧 AI 助手”）要随新信息架构复核。

### 1.4 独立工具与演示页：`outport_calendar.html`、`animation.html`、`resource-add-remove.html`、`timeline-sticky-header.html`

- **日历导出页**：两枚按钮导出“新增”或“全部”日程；先访问旧 `GET /get_calendar/check_modified_events`，再用 `GET /get_calendar/get_outport_calendar` 下载；若发现修改，以 `prompt` 选择包含或忽略。该流程依赖 legacy 端点。
- **重构要求（导出）**：这是必须产品确认的功能，不得静默删除；但旧端点已不属于 V2 Planner。新前端应改用规范化 iCalendar 下载/Feed 能力或为 V2 新建明确导出契约，使用正式 modal 替代 `prompt`。
- **`animation.html`**：全屏 Canvas 粒子/旋涡视觉演示，使用 `requestAnimationFrame`、窗口 resize，无服务端数据；内容注明素材来源。
- **`resource-add-remove.html` 与 `timeline-sticky-header.html`**：均为硬编码 2023 房间/事件的 FullCalendar resource timeline 实验页面；前者可临时新增/点击删除 resource，后者演示 sticky header/scrollbar。无业务 API。
- **重构要求（演示）**：三者不是当前 Planner 产品数据入口；迁移时应明确标为开发演示并移至 Storybook/Playground，或在产品路由中移除。若保留资源排程概念，须另立需求，不可把硬编码 demo 当作日程功能。

### 1.5 认证与账户恢复：`user_login.html`、`user_register.html`、`password_reset.html`、`user_preferences.html`

- **登录页**：Django 服务端 form POST，渲染用户名/密码字段级错误；提供密码找回与返回落地页，条件显示 ICP/公安备案。无前端 API。
- **注册页**：三步流程（填写信息 → 邮箱验证码 → 成功跳转首页），含测试模式提示和开发环境验证码显示；60 秒重发倒计时、Bootstrap Toast、前端密码一致/长度校验。调用 `POST /api/email/request-verification/`、`POST /api/email/verify-code/`，最终以 `FormData` POST `/user_register/`；所有写调用带 CSRF。
- **密码找回页**：两种互斥入口：邮箱验证码三步流程，或“用户名 + API Token”直接重置；调用 `/api/password-reset/request/`、`verify/`、`reset/`、`token/`。使用动态 Bootstrap Toast，成功两秒后跳回登录页。测试环境会在页面显示验证码。
- **偏好页**：编辑“第 n 周起始日期”和“创建日程自动将结束时间设为 DDL”，通过当前页面 POST（jQuery `$.ajax`，`preferences` JSON 字段）保存；有返回首页按钮。
- **重构保留要求**：认证页面可以独立为轻量 React route 或继续由 Django 服务端渲染，但注册、邮件验证、重发限流反馈、两种重置方式、测试模式提示、备案页脚与 CSRF 语义不可遗漏。偏好设置应并入统一 Settings 页面，移除 jQuery、`alert` 与页面相对 URL。
- **安全/体验注意**：验证码及 API Token 输入不得进入持久浏览器状态或日志；测试验证码显示必须仅由服务端显式开发/测试标记决定。Toast 内容必须采用文本渲染，避免把服务端错误原样插入 HTML。

### 1.6 数据、Token 与非产品页面：`user_data.html`、`token_management.html`、`memory.html`、`me.html`、`trial.html`

- **`user_data.html`**：开发/调试用途的“您的数据”页。遍历 `user_data`，将每个 JSON 值做可折叠彩色树渲染；长数组/对象默认折叠。当前通过 `|safe` 注入并以 `innerHTML` 拼接 JSON 字符串，存在 XSS 与 archive 暴露风险。
- **重构要求（数据页）**：不要将 Planner legacy archive 作为普通用户功能暴露。若保留，应是受限管理员诊断页，使用安全 JSON tree 组件、字段脱敏、权限和下载审计。
- **`token_management.html`**：显示用户名、邮箱、用户 ID、当前 Token、复制按钮、安全提示与 HTTP 调用示例；可测试、刷新、删除 Token，调用 `/api/auth/token/verify/`、`refresh/`、`delete/`。页面示例仍写旧 `/api/reminders/`，是必须更正的过时文案。
- **重构要求（Token）**：保留复制/刷新/删除前确认、泄露提醒和 API 示例，但改指向 `/api/v2/`，Token 默认遮罩，复制后短暂反馈；不用 `alert/confirm`。
- **`trial.html`**：面向移动端的试用入口，注册/登录/关于项目三个 CTA、非商业及数据持久性免责文案、备案、亮暗主题、触觉反馈；无业务 API。应和公开 landing 共用品牌组件，而不是复制主题实现。
- **`memory.html`**：与日程产品无关的静态仿站练习页，引用工程外相对图片/字体图标；无 API、无可迁移业务功能。
- **`me.html`**：与日程产品无关的个人履历/作品集，使用 Tailwind CDN、外部随机图片、移动菜单、锚点平滑滚动、IntersectionObserver 入场/技能条动画、返回顶部；“留言”只弹出成功提示，不提交数据。
- **重构取舍**：`memory.html`、`me.html` 应从产品 SPA 中剥离为独立站点/静态示例，不能污染主应用依赖。若保留 `trial`，要求复用公开站组件、主题 token 和无障碍动效策略。

### 1.7 登录后工作台骨架：`core/templates/home.html`（结构与模板内逻辑，第一轮）

- **三栏主界面**：顶部岛式导航；左侧 Todo 面板；中部日程/日历；右侧 Agent 面板。布局宽度由 `PanelResizer` 管理，移动端另有触摸日历支持。
- **顶部全局搜索**：折叠式搜索框，按对象类型、个人日程组、分享组、时间范围过滤，结果以 dropdown 展示。设置菜单提供文件管理、设置、帮助、登出。
- **Todo 面板**：列表/艾森豪威尔四象限切换；五种重要紧急组合筛选、日程组筛选、超量任务 overflow 区；新增 Todo 入口。
- **提醒面板**：创建入口，按时间范围、状态、优先级、类型过滤，提醒卡片列表。
- **日历面板**：创建 Event、管理个人日程组、课程导入、今天/前后导航；“我的日历/分享组”页签；日程过滤（四象限、DDL、有/无重复、是否显示提醒、日程组、分享成员）；FullCalendar 容器。
- **Agent 面板**：上下文用量条、连接状态、新建/历史会话、记忆优化、展开；会话 Todo、会话历史、消息流、工具选择、附件类型/上传/内部内容/云盘选择器、输入框和发送区。
- **Event 表单**：创建与编辑都包含标题、个人组、开始/结束、DDL、描述、四象限、分享组；创建/编辑的完整 RRule UI（频率、间隔、COUNT/UNTIL、星期、按日/按第几周月度规则、预览）。重复编辑有 `single/all/future/from_time` 选择及 future 时间点选择。
- **其他 Modal/设置**：个人信息、对话风格、工作流规则、用户名前/密码修改、个人组/分享组的创建/加入/成员/颜色/编辑/详情、课程导入、Token 管理、Agent 模型/优化/技能/记忆、上下文可视化 UI/JSON overlay。
- **模板内接口**：认证 Token、账户信息、Agent memory、课程导入使用现行 API；但该模板仍有旧 `/get_calendar/events/`、`/api/reminders/`、`/api/todos/`、旧分享组 events、旧用户设置及旧个人组创建调用。这些是当前静态代码遗留，重构时必须统一清除或改为 V2/明确的非 Planner设置接口。
- **初始化与状态**：注入 CSRF、`planner-entrypoints` JSON；实例化 Modal/Event/Todo/Reminder/RRule/Panel/AgentSkills 等全局对象，随后由 `PlannerV2Client` 及各模块加载。模板内还夹杂大量业务 JS（Token、账户、memory、导入、toast），这是拆分为独立 React feature 的首要目标。
- **重构保留要求**：新工作台应保留所有以上可见能力，但按领域拆为 Calendar、Todo、Reminder、Search、Groups、CourseImport、Agent、Settings feature。Modal 状态、筛选状态和异步 loading/error 状态必须成为可测试的组件状态，不再依赖全局 ID 与 inline `onclick`。

#### 已逐段核对：文档骨架、全局搜索与 Event 创建表单（`home.html` 1–1200）

- 页面在模板层同时加载 Bootstrap 5、Font Awesome、四份产品 CSS，并另塞入大块主题覆盖 CSS。包括 TODO 详情、头像/卡片 hover、三类深色主题、原生日期控件、重复设置卡片的定向补丁；重构时应把 token/组件变体收归主题系统，删除“通过 `!important` 强制隐藏 `editEventRecurringInfo`”这类历史补丁。

#### 已逐段核对：工作台其余声明式结构与资源顺序（`home.html` 365–7186）

- 左栏是 Todo（列表/四象限/溢出区、重要/紧急/组筛选）和 Reminder（窗口、状态、优先级、类型、是否重复筛选）；中栏是 Calendar（个人与分享组 tab、课表导入、Event/DDL/重复/Reminder/组/成员筛选）；右栏是 Agent（上下文用量、会话、记忆优化、会话 Todo、消息流、工具、附件、输入）。这三个面板及两条分隔器是工作台的稳定信息架构，重构不应将它们折成单一页面。
- 这里声明了 Event 编辑/重复删除确认、Todo 创建/编辑/详情、Reminder 创建/编辑/详情、Event 详情、Agent 展开、四步课表导入、个人日程组管理、用户设置、个人资料/工作流/账号变更、两套分享组管理和上下文可视化等对话框。两套“分享组管理”和多组重复表单是重构需合并而不是照抄的明显重复。

- 课表导入是 Cookie 输入/帮助、当前或历史学期选择、课程预览与勾选、目标或新建日程组、结果/加载状态的向导。这是独立 feature，不应因迁移主日历而遗漏；Cookie 是敏感输入，迁移时必须做到不落入日志、浏览器持久化或 Agent 附件。
- 设置包含周数/学期与显示偏好、默认日程、提醒、Agent memory/人设/工作流/模型/上下文优化/Token 统计/技能、账户/Token/数据等；应按路由或 tab 延迟加载，避免今天这样一次生成超大 DOM。
- 资源执行顺序高度耦合：Bootstrap/FullCalendar/MathJax、分享组脚本和约 650 行 inline 全局搜索/设置 JS 在前，再按主题、设置、Reminder、RRule、Modal、Group、Planner Client、Event、Todo、手势、Agent、Quick Action 顺序加载，最后初始化。React 重构须把依赖改为显式 import；FullCalendar 及 MathJax 只在需要它们的 feature 懒加载。

#### 已逐行核对：全局搜索与设置 inline 逻辑（`home.html` 4407–5066）

- 搜索 UI 的开合和设置岛互斥；筛选（类型/个人组/分享组/时间）写 `sessionStorage`。搜索先加载 files；对于 normalized cohort，Planner 搜索走 `/api/v2/search/`，限定默认过去 30 天至未来 90 天、`page_size=100`，文件仍本地过滤；旧分支会全量拉取 Event/Reminder/Todo/每个分享组 Event。
- 结果可打开 Event 详情、进入 Reminder 编辑、Todo 详情或在新标签进入文件所属文件夹。现有 V2 搜索没有把个人组、分享组、时间筛选完整送到后端，且将 occurrence/ref 详情丢失后靠缓存/旧格式打开，是重构时要先定义的 `SearchResult` 与跳转行为。
- 设置岛只提供文件管理、设置 modal、帮助、退出登录；使用 `confirm`、DOM order/style 和定时动画。新 AppShell 应由路由/Popover/Dialog 管理，搜索用 debounced query + AbortSignal，且详情导航应带 entity/occurrence identity，不依赖内存搜索数组。

### 1.8 Planner V2 传输层：`planner-v2-client.js`、`planner-v2-scope.js`

- **职责**：读取模板 `planner-entrypoints-data`，请求 `/api/v2/planner/bootstrap/`，在一个页面会话内固定 cohort entrypoint 状态；所有请求强制 `credentials: same-origin`，统一 JSON 解析/错误对象/CSRF JSON options。
- **Calendar projection**：一次并发读取 Event occurrence、Event definition、个人组和 Reminder occurrence；把 normalized 数据转换成 FullCalendar 所需字段，并将 `occurrence_ref`、`source_version`、series/共享信息放入 `extendedProps`。
- **写操作**：封装 Event 创建、按 scope patch/delete Event、按 scope patch/delete Reminder，以及分享日历的 occurrence 查询；负责从 payload 移除 UI 专用字段、将旧 UI 字段归一化。
- **重构保留要求**：保留为 TypeScript API client（建议以 TanStack Query query/mutation 封装）；bootstrap 是 route 进入工作台前的前置数据。任何新组件不得重建 legacy fallback。

#### 已逐行核对的边界（`planner-v2-client.js` 1–252、`planner-v2-scope.js` 1–58）

- `PlannerV2Client` 在构造期即启动 bootstrap；bootstrap 失败时故意进入 fail-closed 的 `blocked` 状态，而非回退 V1。重构时须避免 React StrictMode 造成同一会话重复 bootstrap：应将它变成单例 query / route loader，并在首次请求完成前显示明确 loading 状态。
- 日历读取当前总是四并发请求（Event occurrence、definition、groups、Reminder occurrence）。Event occurrence 自身不带 DDL、共享组和 recurrence 详情，因此前端以 definition map 进行补全；新的 DTO/selector 层要显式表达这种关联，不能把 FullCalendar `extendedProps` 当作唯一事实来源。
- 读取范围采用 ISO 时间参数；后端返回的 occurrence reference（entity、series、recurrence、source version）是后续 mutation 的身份与乐观锁凭据。拖放、编辑、删除不可只传显示 id。
- Event payload 当前默认时区 `Asia/Shanghai`，只在值为 truthy 时带 `ddl_at` / recurrence；重构应把“清空 DDL”“清空重复规则”建模为显式 `null`，避免表单空值在序列化时丢失。
- `buildChanges` 规定重复 Event：单次可变日期；all/future 只能改钟点及时长，禁止跨日；all 会把实例时间差平移回 master。若 recurrence 变更则设置 `override_policy`。这是一项业务规则，不得被通用表单库或日历拖拽默认行为绕开。
- Reminder scope 会把 UI 旧名称收敛为 `single` / `this_and_future`；但 P3 的产品限制仍应由 UI capability（而非这个低层函数）拦截。新的 API 层应将后端返回的 capability/错误转化为可读禁用理由。

### 1.9 日程与日历交互：`event-manager.js`

- **日历生命周期**：初始化 FullCalendar、多视图/首屏滚动位置/可选区、事件源刷新及滚动可用性处理；合并个人 Event、分享组 Event 和 Reminder 展示。
- **创建与转换**：日历选区创建 Event；Todo 可拖放进日历，解析预计时长/默认时长后转换为 Event 并预填表单。
- **筛选与呈现**：按四象限、DDL、有无重复、提醒、个人组、分享成员过滤；按个人/分享、日程组、成员色、视图类型做不同事件外观和共享圆点；有 Event/Reminder 状态、优先级、逾期颜色和图标映射。
- **编辑与重复**：拖拽/resize/编辑/删除有单次、全部、此后/从指定实例 scope 选择；包含未来实例选择、规则变更按钮、字段禁用/提示、bulk operation 和 `override_policy`/scope change 构造。
- **Reminder 联动**：点击日历中 Reminder 可看详情、完成/忽略/恢复/延后（预设或自定义）、编辑、删除；日历刷新后同步 Reminder 视图。
- **遗留风险**：模块中仍大量调用旧 Event/Reminder API，V2 只覆盖部分路径；重构需把该功能完全切到 `PlannerV2Client`，不能逐段照搬。特别是拖拽、重复范围、Reminder 状态 action 和共享日历必须先写契约测试再实现。

#### 已逐段核对：FullCalendar 配置、mutation、详情与外观（`event-manager.js` 1–3909）

- FullCalendar 是日历渲染内核：中文、月/周/日/列表/自定义 2 日视图、周数、全天、当前时间、默认滚动、可编辑/可选；当前 view 与日期写入 Settings。现代化应继续使用其 React 绑定（或同等成熟日历），并把 `datesSet`、事件源、loading/error、视图首选项作为明确的状态接口。
- 数据源为可见窗口，同一窗口请求会共享 Promise 避免竞态；V2 正确路径走 `plannerV2Client.fetchCalendar`，合并 Event 与 Reminder。筛选包含 Event 四象限、是否 DDL、是否重复、组、分享组成员，以及 Reminder 显示开关；群组视图只把部分筛选应用于自己的日程。这些是产品规则，需在新 selector 上逐条回归测试。
- Todo 拖到日历仅预填 Event 创建框（默认当前+1h，时长取 Todo 预计值，保留组/DDL/四象限），最终创建和 Todo→Event 的真正转化是不同操作；迁移时必须避免把“预填新 Event”误做成“转化并删除 Todo”。Event 或 Reminder 从 FullCalendar 拖到 Agent 也以 occurrence ref 作为内部附件身份。
- 拖动/resize Event 先限制结束不晚于 DDL；重复 Event 会选 single/future/all。V2 对 future/all 不许改日期，仅可改钟点/时长；all 会以 master start 计算偏移。创建、删除、bulk edit 都通过 occurrence ref + source version；这套范围规则需要作为日历集成测试的关键矩阵。
- Event 详情提供时间、拥有者、组、四象限、描述、DDL、RRULE；共享只读日程隐藏编辑。Reminder 详情有状态与延后动作，日历打开对话框时禁用 select。新 UI 要保持只读权限、详情展示与 focus/keyboard 正确性。
- 共享视觉不能遗漏：个人被分享 Event 显示分享组彩点；分享日历中本人日程可显示组色/成员色双色，其他成员以成员色和斜纹或图标表示；月/时间网格/列表各有投影规则。应将它们收敛为 Calendar event renderer，而非散落 DOM style 写入。
- 主要技术债：大约 3,900 行混合 FullCalendar 回调、动态 Bootstrap dialog、inline action、V1 fallback 和 DOM style；`from_time` 只从当前可见窗口找 anchor，窗口外不可用。React 实现应避免使用可见实例列表作为唯一的范围 anchor，并通过 V2 occurrence 查询或后端能力决定范围操作。

### 1.10 Todo：`todo-manager.js`

- **展示**：常规卡片列表与艾森豪威尔四象限两种视图；优先级/紧急度图标、截止时间（相对日期）、日程组、描述、状态；空间不足时进入 overflow 区。
- **交互**：创建、详情、编辑、删除、拖拽排序/拖入四象限、拖入日历转换为 Event；过滤器保存并动态加载个人日程组。
- **状态**：本地 `todos`、过滤条件、视图模式；使用 HTML 转义后拼接卡片。
- **接口/风险**：CRUD、转换、拖拽更新仍调用旧 `/api/todos/*` 路径（转换路径还与现行 V2 不一致）。重构必须用 `/api/v2/todos/` 和 `/convert/`，每次 mutation 用 version 返回值刷新缓存。

#### 已逐段核对：加载、详情、转换与四象限交互（`todo-manager.js` 1–1450）

- 读取只把 `status` 交给服务端，其余重要/紧急、组筛选与 due 排序在浏览器执行；V2 DTO 被即时改名为旧 `id/groupID/due_date/estimated_duration`。建议迁移时保留 API DTO 原样，另写 selector；筛选条件应成为 query key 的一部分或清晰的 client selector，不能混用两种命名。
- 卡片支持组颜色、四象限 emoji、相对截止文案、描述、预计时长、编辑/删除、详情；详情依据字段有无显示/隐藏行，并锁住 `body` 滚动。新的 Dialog 需要可访问的焦点捕获和关闭恢复，不能继续手动写 `body.style.overflow`。
- 所有 Todo 卡（含四象限精简卡）可拖到 Agent，DataTransfer 中带内部元素类型、id、标题、描述、日程组、截止/耗时、重要紧急；这正是 Agent 内部附件的一个数据入口，后续应统一为类型安全的 `InternalAttachment`，并确认拖放与点击/触摸不冲突。
- Todo→Event 使用 V2 `/api/v2/todos/{id}/convert/`，会使用 Todo 的 `version` 和 Event payload，成功后刷新 Todo 与日历；这是要保留的原子跨实体操作。不能改为两个浏览器 mutation。
- 四象限切换会持久化，按左栏宽度在横向/纵向排布，利用平方根权重动态设置 CSS Grid 行列，容量不够的任务显示到 overflow；拖到任一象限会 PATCH importance/urgency（包括“其他”清空两值）。这不是通用看板，重构应选择支持可访问拖放的库，并保留动态面积/overflow 这一特有视觉规则。
- 风险：代码仍有 legacy 分支和 inline onclick；V2 拖拽成功后本地把 `version += 1`，而非采用服务端返回对象，可能在并发下陈旧。新实现统一 mutation 成功后的失效/返回 DTO 更新，且将 confirmation/toast/error 纳入 mutation 状态。

### 1.11 Reminder：`reminder-manager.js`

- **展示与筛选**：按窗口、状态、优先级、类型呈现 Reminder 卡；详情含状态按钮、延后菜单、触发时间和重复信息。
- **提醒交付 UX**：定时检查即将到期和提前提醒；优先尝试浏览器 Notification，失败回退页面内通知/alert；可标记已通知。
- **重复编辑**：创建/编辑/删除、RRule 变更确认、单次或批量范围、将重复提醒转单次、预设/自定义 snooze；包含 RRule 的构建、解析、预览和编辑 scope UI。
- **现行限制**：代码含 V2 action/definition 兼容辅助，但大部分 CRUD/bulk/status/notification 仍是已封存 Reminder V1。新版产品只允许重复 Reminder `all` 内容编辑和 occurrence action 单次状态；新 UI 必须明确禁用未开放范围，而不是执行旧逻辑。

#### 已逐段核对：definition/occurrence 投影、通知、编辑与状态（`reminder-manager.js` 1–2025）

- 侧栏先取 definition，再按筛选时间窗取 occurrence，并拼成卡片；“重复”不是一个汇总卡，而是每个 occurrence 一张卡。这解释了筛选与日历的一致性要求：新 UI 要按 occurrence 窗口查询并分页/虚拟化，不能只读取 definition 后手算 RRule。
- 当前筛选时间为 today/week/month/quarter/year/all；状态支持 active/completed/dismissed/snoozed，类型支持 single/recurring/detached。类型判断依赖 `FREQ=`；新 DTO 层应使用 recurrence 是否存在，而非解析文本。
- 提醒卡能拖进 Agent（仅携带 title、definition id、occurrence ref），有已过期显示、优先级、内容、提前提醒、完成/忽略切换、延后（15m/1h/1d/自定义）、编辑/删除。内部附件必须保留 occurrence ref，才能让 Agent 得到用户拖入的是哪次实例。
- 本地提醒每分钟遍历已加载实例；主提醒与提前提醒会显示浏览器 Notification 和页面内提示，随后 `mark_sent`。这受当前筛选时间窗影响，因此 UI 重构时应明确“仅为已加载时间窗触发”的现状，或单独设计可靠的 server/push 策略；绝不可误宣称浏览器打开时就会触发所有未来提醒。
- V2 创建支持 title/content/priority/status/trigger/recurrence；all 编辑从 definition 构造 reference 并用 `patchReminder(..., 'all')`，可把 single 改成重复、也可 all 取消 recurrence。重复实例的完成/忽略/延后/重置经 `/api/v2/reminders/occurrences/action/`，且携带 occurrence ref 和 source version。
- 旧 UI 仍生成“单次/全部/此后/指定时间”批量弹窗，并保留 old V1 的 transform/convert 分支；它和 P3 产品能力冲突。新组件不得把这些选项作为可执行能力：重复 Reminder 的定义编辑明确只显示 all；occurrence 的 action 明确是单次。其他 scope 应展示解释性禁用项或不展示。
- 风险：deleteSingleReminder 的 V2 分支没有按 occurrence ref 删除，旧 bulk UI 在 V2 下会传未被后端支持的 scope；自定义 snooze 使用 `prompt`；多个 dynamic modal 混用 Bootstrap 与手写显示。迁移需以 React Dialog、datetime input 和 capability-based action menu 一次解决，并加 occurrence/definition/并发 version 契约测试。

### 1.12 个人日程组与 RRule 表单：`group-manager.js`、`rrule-manager.js`、`modal-manager.js`

- **GroupManager**：个人 EventGroup 列表、创建、编辑、删除多面板 modal，刷新所有 Event/Todo 表单下拉组。当前 create/update/delete 使用旧接口，重新加载才部分支持 V2。

- **ModalManager**：全局 modal 生命周期、表单监听/校验/清理、Event/Todo/Reminder 创建/编辑/删除，DDL 与结束时间同步，重复 Event scope/自定义日期时间控件，RRule 解析与修改模式，四象限选择、字段序列化和 toast。

#### 已逐段核对：日程组 CRUD 与表单职责（`group-manager.js` 1–441；其余已按方法分段清点）

- 日程组是个人 EventGroup（不是分享组），具有名称、描述、颜色、version。删除可选择只删组或同时删关联 items；成功后刷新 Calendar、Todo 与所有 Event/Todo 表单的组下拉。这个级联选项及刷新范围是迁移验收项。

### 1.14 分享组：`share-groups.js`

- **历史与回滚**：从 `/api/agent/history/` 恢复消息、工具调用、附件、summary 和 rollback window；会话列表支持新建/切换/重命名/删除。按用户消息 index 回滚到 `/api/agent/rollback/to-message/`，成功后重建历史并刷新 Planner。
- **消息 UI**：Markdown/代码、思考/推理块、工具折叠结果、欢迎/typing/Toast、消息 modal、会话级 Todo 面板、上下文用量条和上下文可视化（UI/JSON、父子 snapshot、token 堆叠条）。
- **附件**：文件上传、云盘树多选、内部 Event/Todo/Reminder 附件、附件格式化、图片/文档预览、选中 tile/移除、回滚后消息附件恢复；对当前模型可用性和待 OCR 内容有检查/提示。
- **重构保留要求**：这是最复杂的 feature，必须拆为 chat transport、session/history、message renderer、tool activity、attachment picker、rollback、context inspector 六个以上组件/状态域。需要现成 Agent UI 但必须能插入本项目 WebSocket 事件、附件 tile 与 rollback window 语义。

#### 已逐段核对：transport、回滚、会话与附件（`agent-chat.js` 1–6201；重点 1–1260、2499–3653、4204–5028）

- Agent 的传输是项目自定义 WebSocket 协议而非标准 AI SDK 流：message/stop/ping、stream_start/chunk/end、reasoning、tool_call/result、finished、action preview/executed、error、quota、recursion、summary、自动命名、上下文快照等事件。候选 Agent UI 必须允许自定义 transport/event reducer；不能因采用库而把后端改为不必要的 SSE 协议。
- 会话 id 与工具偏好/草稿/流状态/递归限制/上下文缓存写 localStorage，且 session id 与 user id 绑定检查；断线最多重连 5 次，工具选择更改会重连。新架构应明确哪些状态可本地恢复（草稿/临时流 UI），哪些以服务端为准（历史、工具、附件、回滚窗口）。
- 回滚窗口在加载会话时建立或复用，切换/新建会话时关闭当前窗口；消息历史的 `can_rollback` 控制按钮。回滚成功后服务端返回 requeued attachment DTO，前端恢复输入和附件，刷新 Planner。这种“仅当前活跃会话、自进入后窗口内消息可回滚”的语义必须保留，且不可仅靠前端隐藏按钮保障。
- 消息渲染包括用户/Agent/错误、Markdown+代码+数学、图片与文档附件 tile/预览、reasoning 折叠、工具进行中/结果折叠、typing、欢迎快捷提示、上下文总结分隔线、会话级 Todo；存在独立展开对话框。新实现需把不可信文本/Markdown/syntax highlight 的 sanitization 与可访问折叠/预览纳入组件选择。
- 附件有四类内部对象（Event/Todo/Reminder/Workflow）、本机上传（20MB、图片/PDF/Office）、云盘多选树，以及已选 tag/预览/移除。内部对象在发送前以 `/api/agent/attachments/internal/` 转为 SessionAttachment，随后所有附件以 `attachment_ids` 经 WebSocket 发送；上传/云盘已直接得到 SessionAttachment。此“先创建可服务端验证的 attachment，再发送 id”是 Agent 能看见附件和回滚能恢复附件的关键合同。
- 当前实现有明显的 DTO 不一致风险：附件去重仅按 type+id，文档预览有时用 `sa_id`、有时用 `id`，内部详情从当前面板缓存查找而非按 occurrence ref 查询；重构时应定义单一 `PendingAttachment` / `SessionAttachment` 联合类型和稳定 key（`sa_id` 优先，内部 object 包含 occurrence ref）。
- 这份 6201 行单文件还混入会话 Todo、工具选择、memory 优化、上下文 token/snapshot JSON/UI inspector、所有 DOM 生命周期。建议选择“可替换 UI 层”的 Agent library，只接管消息/Composer/工具呈现；WebSocket、rollback、attachment 和 session resource 仍由本项目 TypeScript client 掌控。

### 2.4 Server state 与前端内部状态

- **TanStack Query** 作为所有 Django HTTP query/mutation 的 server-state 层。query key 必须包含用户域、资源、可见时间窗、分享组和筛选器；mutation 成功后按资源精确 invalidation 或使用服务端返回 DTO 更新，而不是直接把本地 version `+1`。其文档说明 query key 是缓存分区核心。[TanStack Query Query Keys](https://tanstack.com/query/v4/docs/framework/react/guides/query-keys)
- **Zustand（仅建议）** 用于不会落入 HTTP cache 的 UI state：三栏临时尺寸、当前 dialog、全局搜索展开、Agent WebSocket reducer、客户端拖放预览。它不是 Planner/文件/会话真相来源；是否纳入最终依赖可在脚手架阶段确认。
- 统一实现 `apiClient`：same-origin credentials、CSRF、JSON 错误、版本冲突 error code、AbortSignal；Planner V2 的 bootstrap 是工作台 route loader 的前置条件。禁止出现任何 legacy URL fallback。

## 3. 暂定推荐架构（供下一轮确认，不开始实施）

**推荐组合：React 19 + TypeScript + Vite，Django 继续提供登录页、同源 API/WS 与生产静态资源；Tailwind + shadcn/ui；TanStack Query；FullCalendar React；assistant-ui（外部状态 runtime）；dnd-kit；react-resizable-panels；Motion；React Hook Form + Zod。**

理由不是“库越多越好”，而是每一项都有当前代码中明确的替换目标：FullCalendar 保护日历行为，assistant-ui 只替换对话 UI，dnd-kit 处理无障碍跨面板拖放，Radix/shadcn 终结手写 modal，Query 保证 V2 version/缓存正确性。其余业务规则仍属于本项目 TypeScript domain 层。

```text
Django templates（登录/公共页，可后迁）
             │  提供静态 SPA 容器、CSRF、/api/*、/ws/agent/
             ▼
React 工作台（按 feature 懒加载）
 ├─ planner-api / TanStack Query ── Event / Todo / Reminder / Group / Share
 ├─ Calendar feature ── FullCalendar React + V2 occurrence projection
 ├─ Agent feature ── WebSocket reducer + assistant-ui external runtime
 ├─ File feature ── files API + shared Attachment picker
 ├─ Settings / Search / Quick Action
 └─ design system ── shadcn/Radix + Tailwind tokens + Motion
```

### 明确不在本阶段引入

- 不迁移回 V1 或兼容 legacy Planner URL；P6 已封存，重构页面只允许 V2 client。
- 不为 Agent UI 改造服务端为 Vercel AI SDK、SSE 或 CopilotKit runtime；先用 adapter 消化当前 WebSocket 事件。
- 不把 recurrence 展开、范围编辑、回滚逆操作放到前端推导；前端只保存 occurrence ref/version 并调用后端能力。
- 不将 File picker、日历内部附件、回滚后的附件恢复各写一份；三者共用 attachment DTO、repository 和预览组件。
