# 思考模式（Reasoning / Thinking）适配升级方案

> 适配范围：DeepSeek V4 Pro/Flash、Kimi K2.6（默认开启思考，可禁用）。
> 不适配：MiniMax-M2.1（锁定为非思考）、用户自定义模型（暂不处理）。

---

## 一、问题根因

LLM 报错示例：

```
Error code: 400 - The reasoning_content in the thinking mode must be passed back to the API.
Error code: 400 - thinking is enabled but reasoning_content is missing in assistant tool call message at index N
```

DeepSeek / Kimi 文档共同规则：

- 思考模式开启时，凡是 `assistant` 消息**带 `tool_calls`**，下一次请求必须把
  `reasoning_content` 一并回传，否则 400。
- DeepSeek：未带工具调用的 `assistant` 消息可丢 `reasoning_content`（API 自动忽略）。
- Kimi K2.6：可通过 `thinking.keep` 控制是否保留历史思考；本方案统一传 `keep="all"`，
  便于实现统一逻辑（多步工具调用必须保留）。

项目当前问题：

1. `_build_chat_llm` 直接构造 `ChatOpenAI`，未传 `extra_body.thinking`；模型默认开启
   思考后返回 `reasoning_content`，但 LangChain 默认不在下一次请求里回传该字段。
2. 历史 checkpoint 中的 `AIMessage` 完全没有 `reasoning_content`，对未来思考模式不兼容。
3. 没有"思考"开关、没有可行性校验、没有响应解析、没有前端渲染。

---

## 二、升级目标

1. 系统模型新增 `supports_thinking / thinking_param_style / thinking_min_max_tokens`
   等元数据字段，由 `api_keys.json` 声明。
2. 用户级"思考开关"独立存储（与模型选择并列），存于 `UserData.agent_config.thinking_enabled`。
3. 切换模型/开关时做可行性校验（不支持思考的模型不允许 `thinking_enabled=True`）。
4. `_build_chat_llm` 根据"模型 + 思考开关"自动注入 `thinking` / `max_tokens` / `temperature` 等参数；
   并通过 `ReasoningAwareChatOpenAI` 子类把 `additional_kwargs.reasoning_content` 注入请求体。
5. 响应侧：从 raw response 提取 `reasoning_content` 写入 `AIMessage.additional_kwargs`，
   随 LangGraph checkpoint 持久化；同时 WS 推一条 `reasoning` 类型消息给前端。
6. 上下文优化：Token 计算把 `reasoning_content` 计入；总结时摘要保留前 200 字思考。
7. 历史兼容：任何思考模式 400 报错触发"静默降级"——本轮重试为 `thinking_enabled=False`，
   不污染配置；前端给一次温和提示。
8. 前端：思考开关 UI（带可行性禁用）、reasoning 块默认折叠、上下文可视化标注思考块、
   历史对话重新渲染思考块。

---

## 三、整体设计原则

- **配置驱动**：模型能力以 `api_keys.json` 字段为准，代码不内置硬编码 model_name。
- **闭环**：响应解析 → checkpoint 持久化 → 下次请求注入，全链路能保证 `reasoning_content`
  与对应 `tool_calls` 一一对应。
- **降级而非补丁**：旧会话不需要打补丁，靠"出错重试"自动落到非思考模式即可。
- **不影响流式**：保持当前节点级流式；reasoning 整段一次性 send 即可，不做 token 流。
