# 认证与权限规范

> 本文档描述 UniSchedulerSuper 后端各种认证方式的使用规范。

---

## 1. 认证体系总览

系统支持三种认证方式，分场景使用：

| 认证方式 | 适用场景 | 实现 |
|---------|---------|------|
| Session 认证 | 浏览器页面（Cookie） | Django 内置 Session |
| Token 认证 | REST API 客户端、移动端 | DRF `TokenAuthentication` |
| Basic Auth | CalDAV 客户端（iOS/macOS/DAVx5） | `caldav_service/auth.py` |

DRF 全局配置（`settings.py`）：
```python
REST_FRAMEWORK = {
    'DEFAULT_AUTHENTICATION_CLASSES': [
        'rest_framework.authentication.TokenAuthentication',
        'rest_framework.authentication.SessionAuthentication',
    ],
    'DEFAULT_PERMISSION_CLASSES': [
        'rest_framework.permissions.IsAuthenticated',
    ],
}
```

## 2. Session 认证（页面视图）

### 2.1 使用方式

页面视图使用 `@login_required` 装饰器；未登录时自动跳转到 `settings.LOGIN_URL = '/user_login'`：

```python
from django.contrib.auth.decorators import login_required

@login_required
def home(request):
    return render(request, 'home.html', {...})
```

### 2.2 登录 / 登出视图

```python
# 登录（表单 POST）
POST /user_login/
Body: username=..., password=...    # form-encoded

# 登出
GET /user_logout/
```

登录成功后重定向到 `/home/`；登出后重定向到 `/user_login/`。

### 2.3 密码策略

Django 内置 `AUTH_PASSWORD_VALIDATORS` 已启用：
- 不能与用户名过于相似
- 最少 8 个字符
- 不能是常用密码
- 不能纯数字

## 3. Token 认证（REST API）

### 3.1 Token 获取流程

```
POST /api/auth/login/
Body: {"username": "...", "password": "..."}
Response: {"token": "9944b09...", "user_id": 1, "username": "admin"}
```

### 3.2 API 请求携带 Token

```http
GET /api/agent/sessions/
Authorization: Token 9944b09199c62bcf9418ad846dd0e4bbdfc6ee4b
```

### 3.3 Token 管理 API

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/auth/login/` | POST | 用户名密码换 Token |
| `/api/auth/logout/` | POST | 删除 Token（登出） |
| `/api/auth/token/` | GET | 获取当前用户 Token（需已登录） |
| `/api/auth/token/refresh/` | POST | 刷新 Token（删旧建新） |
| `/api/auth/token/verify/` | GET | 验证 Token 有效性 |
| `/api/auth/token/delete/` | DELETE | 删除 Token |

### 3.4 视图中访问当前用户

```python
@api_view(['GET'])
@permission_classes([IsAuthenticated])
def my_view(request):
    user = request.user   # 已认证的 User 对象
    ...
```

## 4. CalDAV Basic Auth

CalDAV 客户端（iOS 日历、macOS 日历、DAVx5、Thunderbird）使用 HTTP Basic Auth。

### 4.1 密码字段支持两种方式

1. **API Token 作为密码**（推荐）：`password` = DRF Token key
2. **明文密码**：传统用户名+密码

优先级：先尝试 API Token，失败后回退到明文密码。

### 4.2 认证入口

`caldav_service/auth.py` 中 `get_user_from_request(request)` 统一处理：
1. 解析 `Authorization: Basic <base64>` 头
2. 尝试 Token 认证：`Token.objects.get(key=password)` 且 `token_obj.user.username == username`
3. 失败则尝试 `django.contrib.auth.authenticate(username, password)`
4. 均失败返回 `None`，CalDAV 视图返回 `401 WWW-Authenticate: Basic`

### 4.3 CalDAV 连接信息

```
服务器地址：https://your-domain/caldav/
用户名：Django 用户名
密码：DRF Token（推荐）或 Django 明文密码
```

## 5. 权限控制

### 5.1 API 视图

- 默认：`@permission_classes([IsAuthenticated])` — 必须登录
- 公开接口：`@permission_classes([AllowAny])` — 如 `api_login`、`speech_to_text`
- 管理接口：暂无独立 Role，均以用户隔离数据（多租户通过 `request.user` 过滤）

### 5.2 数据隔离

所有业务数据通过 `UserData.objects.filter(user=request.user)` 或  
`AgentSession.objects.filter(user=request.user)` 隔离，**禁止**查询全量数据后在应用层过滤。

### 5.3 日历订阅（URL Token）

日历订阅接口不支持请求头认证（Apple Calendar 限制），使用 URL 中的 token 参数：

```
GET /api/calendar/feed/?token=<drf-token>&type=all
```

视图中通过 `token` 参数查找 `Token.objects.get(key=token_str).user`，  
**此接口不应用 `@permission_classes`，需在视图内手动验证 token**。

## 6. CSRF 保护

- 所有页面表单提交受 Django CSRF 中间件保护。
- REST API（`@api_view`）通过 DRF SessionAuthentication 自动执行 CSRF 检查。
- 公开 API 登录接口使用 `@csrf_exempt` 豁免（Token 认证无需 CSRF）。
- 前端 `fetch` POST 请求必须携带 `X-CSRFToken` 头（见前端规范）。

受信任来源配置（`settings.py`）：
```python
CSRF_TRUSTED_ORIGINS = [
    'https://unischedulersuper.online',
    'https://www.unischedulersuper.online',
]
```
