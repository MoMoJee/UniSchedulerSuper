# 日志与错误处理规范

> 本文档描述 UniSchedulerSuper 后端日志记录和错误处理的约定。

---

## 1. 统一 Logger

### 1.1 导入方式

所有模块统一从根目录 `logger.py` 导入：

```python
from logger import logger
```

**禁止**：
```python
# ❌ 禁止这样做
import logging
logger = logging.getLogger(__name__)
```

> `logger.py` 中的 `ProjectLogger` 是单例，封装了 `ConcurrentRotatingFileHandler` 多进程安全写入。  
> 直接使用 `logging.getLogger()` 会绕过统一格式化配置，在多进程（Daphne/ASGI）下存在文件写入竞态。

### 1.2 日志级别使用规范

| 级别 | 方法 | 使用场景 |
|------|------|---------|
| DEBUG | `logger.debug()` | 开发调试，不应出现在生产代码中 |
| INFO | `logger.info()` | 正常业务操作记录（登录、创建、删除等） |
| WARNING | `logger.warning()` | 可能有问题但不影响功能（Key 未定义、降级处理） |
| ERROR | `logger.error()` | 操作失败，需人工关注 |

### 1.3 日志格式

输出格式（`settings.py` 中配置）：
```
2026-04-27 09:30:00 | INFO     | ProjectLogger | User admin logged in via API
```

日志文件：`logs/application.log`  
滚动策略：单文件最大 2.5MB，保留 10 个备份（`maxBytes=5*512*1024, backupCount=10`）

## 2. 日志内容规范

### 2.1 必须记录 INFO 的操作

- 用户登录 / 登出
- Token 创建 / 删除 / 刷新
- 任何涉及数据变更的 Agent 操作
- CalDAV PUT / DELETE 操作

```python
logger.info(f"User {username} logged in via API, token {'created' if created else 'retrieved'}")
logger.info(f"[CalDAV] DELETE event {event_uid} for user {user.username}")
```

### 2.2 必须记录 ERROR 的场景

```python
try:
    ...
except Exception as e:
    logger.error(f"API login error: {str(e)}")
    return Response({'error': f'操作失败: {str(e)}'}, status=500)
```

**规则**：
- `except Exception as e` 块中必须有 `logger.error()`，绝不 `except: pass`。
- 错误信息包含操作名和 `str(e)`，便于定位。
- 不要把完整 traceback 暴露到响应体，仅记录到日志。
- 需要 traceback 时使用 `logger.error("...", exc_info=True)`，不要手动拼接 `traceback.format_exc()`。

### 2.3 CalDAV 调试日志

CalDAV 请求使用 `logger.debug()` 记录详细信息（认证类型、请求路径等），  
生产环境不影响性能，开发时可调整级别查看：

```python
logger.debug(f"[CalDAV Auth] Authorization header type: {auth_type}")
logger.debug(f"[CalDAV] PROPFIND {request.path}")
```

## 3. 请求日志中间件

`core/middleware/request_logger.py` 的 `RequestLogMiddleware` 自动记录所有 HTTP 请求：

```
203.0.113.1 - admin - "POST /api/events/bulk-edit/" 200 1234 0.045s
```

格式：`{IP} - {username} - "{METHOD} {PATH}" {STATUS} {Content-Length} {duration}s`

该中间件在 `settings.py` 中排在所有中间件**最前面**，确保记录完整时间：
```python
MIDDLEWARE = [
    'core.middleware.request_logger.RequestLogMiddleware',  # 第一个
    'caldav_service.middleware.CalDAVRoutingMiddleware',
    ...
]
```

## 4. 错误响应规范

### 4.1 标准错误结构

```python
# DRF Response（agent_service 等新接口）
return Response({'error': '参数错误: title 不能为空'}, status=400)

# JsonResponse（core 旧接口）
return JsonResponse({'status': 'error', 'message': '...'}, status=400)
```

### 4.2 异常处理模板

```python
@api_view(['POST'])
@permission_classes([IsAuthenticated])
def some_api(request):
    try:
        # 业务逻辑
        result = do_something(request)
        return Response({'data': result})

    except ValueError as e:
        # 可预期的参数错误
        return Response({'error': str(e)}, status=400)

    except Exception as e:
        # 未预期的错误
        logger.error(f"some_api error for user {request.user.username}: {str(e)}")
        return Response(
            {'error': f'操作失败: {str(e)}'},
            status=status.HTTP_500_INTERNAL_SERVER_ERROR
        )
```

### 4.3 非关键失败的处理

对于不影响主操作成功的副作用（如分享组同步、Agent 日志写入），  
失败时记录 WARNING/ERROR 但**不中断**主操作：

```python
try:
    sync_group_calendar_data(groups, user)
except Exception as e:
    logger.error(f"同步分享组数据失败: {str(e)}")
# 继续返回成功
return Response({'event': event})
```

## 5. 安全日志注意事项

- **禁止**在日志中记录完整密码、API Key、Token 值。
- 记录 Token 操作时只记录"created/retrieved/deleted"，不记录 Key 本身。
- 记录用户操作时只记录 `username`，不记录 `user_id`（用于匿名化）。

```python
# ✅ 正确
logger.info(f"User {username} refreshed token via API")

# ❌ 禁止
logger.info(f"New token: {new_token.key}")
```
