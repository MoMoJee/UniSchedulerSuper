# 数据模型规范

> 本文档描述 UniSchedulerSuper 后端数据存储模型约定。

---

## 1. 核心存储模型：`UserData`

### 1.1 模型定义

`UserData`（位于 `core/models.py`）是全部用户业务数据的存储载体，采用 **Key-Value + JSON** 结构：

```python
class UserData(models.Model):
    user  = models.ForeignKey(User, on_delete=models.CASCADE)
    key   = models.CharField(max_length=100)   # 数据类型标识
    value = models.TextField()                  # JSON 序列化的数据
```

- 每个 `(user, key)` 组合唯一。
- `value` 存储的是 Python 对象序列化后的 JSON 字符串。
- 禁止直接操作 `value` 字段，必须通过 `get_value()` / `set_value()` 方法读写。

### 1.2 标准 Key 列表

所有合法 key 必须在 `DATA_SCHEMA` 中预定义，禁止使用未定义的 key：

| Key | 类型 | 说明 |
|-----|------|------|
| `events` | `list` | 日程列表（支持 RRule 重复事件） |
| `todos` | `list` | 待办列表 |
| `reminders` | `list` | 提醒列表 |
| `events_groups` | `list` | 日历分组 |
| `user_settings` | `dict` | 用户业务偏好设置 |
| `user_interface_settings` | `dict` | UI 显示设置（视图、布局、过滤器） |
| `ai_chatting` | `dict` | Token 余额、昵称 |
| `planner` | `list` | AI Agent 操作日志 |

### 1.3 读写 API

**获取数据（推荐用法）：**
```python
user_data, created, result = UserData.get_or_initialize(
    request=request,
    new_key="events",
    data=[]      # 可选，若 key 不存在时写入的初始值
)
if user_data is None:
    return JsonResponse({'status': 'error', 'message': 'Failed to initialize'}, status=500)

events = user_data.get_value() or []
```

**写入数据（必须包 reversion）：**
```python
import reversion

with reversion.create_revision():
    reversion.set_user(request.user)
    reversion.set_comment("Create event: 标题")
    user_data.set_value(events)
    user_data.save()
```

## 2. DATA_SCHEMA 约定

`DATA_SCHEMA` 定义每个 key 的结构、类型和默认值，是数据模型的"源头真相"。

### 2.1 字段定义格式

```python
"field_name": {
    "type": str,          # Python 类型（str / int / float / bool / list / dict）
    "nullable": True,     # 是否允许 None
    "default": "",        # 默认值（dict 类型只写 {}，不嵌套）
    "comment": "说明",    # 可选注释
}
```

> **重要**：`dict` 类型的字段 `default` 只写 `{}`，嵌套结构通过 `"items"` 子字段描述。  
> 这是防止 Python 可变默认值 bug 的约定。

### 2.2 枚举值约定

| 字段 | 合法值 |
|------|--------|
| `events[].importance` | `"important"` / `"not-important"` / `""` |
| `events[].urgency` | `"urgent"` / `"not-urgent"` / `""` |
| `events[].status` | `"confirmed"` / `"tentative"` / `"cancelled"` |
| `todos[].status` | `"pending"` / `"in-progress"` / `"completed"` / `"cancelled"` |
| `reminders[].status` | `"active"` / `"dismissed"` / `"snoozed"` |
| `reminders[].priority` | `"critical"` / `"high"` / `"normal"` / `"low"` / `"debug"` |
| `events_groups[].type` | `"work"` / `"personal"` / `"study"` / `"health"` / `"social"` / `"other"` |

## 3. 重复事件（RRule）字段

重复事件在 `events[]` 中通过以下字段管理：

| 字段 | 类型 | 说明 |
|------|------|------|
| `rrule` | `str` | RRule 字符串，如 `FREQ=WEEKLY;BYDAY=MO,WE` |
| `is_recurring` | `bool` | 是否为重复事件系列成员 |
| `is_main_event` | `bool` | 是否为系列主事件（模板）|
| `is_detached` | `bool` | 是否为脱离编辑的独立实例 |
| `series_id` | `str` | 系列唯一 ID（同系列共享） |
| `recurrence_id` | `str` | 格式 `%Y%m%dT%H%M%S`，标识原始时间槽 |
| `parent_event_id` | `str` | 指向主事件的 `id` |
| `original_series_id` | `str` | CalDAV "此及以后"截断时的原始系列 ID |

**规则：**
- 每个系列有且仅有一个 `is_main_event=True` 的主事件。
- 脱离编辑（单次修改）的实例：`is_detached=True`，保留 `series_id`。
- 删除系列主事件时同步删除所有 `series_id` 相同的实例。
- RRule 字符串末尾不加分号（写入前执行 `.strip().rstrip(';')`）。

## 4. 时间格式规范

| 场景 | 格式 | 示例 |
|------|------|------|
| `events[].start` / `.end` | `YYYY-MM-DDTHH:MM`（本地时间，无时区） | `2026-04-27T09:00` |
| `todos[].due_date` | `YYYY-MM-DDTHH:MM` 或 `YYYY-MM-DD` | `2026-04-30` |
| `reminders[].trigger_time` | `YYYY-MM-DDTHH:MM` | `2026-04-27T08:00` |
| `last_modified` | `YYYY-MM-DD HH:MM:SS` | `2026-04-27 09:30:00` |
| `recurrence_id` | `%Y%m%dT%H%M%S` | `20260427T090000` |

> **注意**：后端存储和 API 均使用**本地时间（东八区 UTC+8）**，不带 `Z`。  
> `EventService._convert_time_format()` 负责将带 `Z` 的 UTC 时间转换为本地时间。

---

## 5. ID 规范

- 所有业务实体（event / todo / reminder / events_group）的 `id` 均为 **UUID 字符串**。
- 生成方式：`str(uuid.uuid4())`。
- `series_id` 同样为 UUID 字符串。
- **禁止**使用自增整数作为业务 ID。

---

## 6. Django ORM 模型（非 UserData）

对于需要独立查询、关联、索引的数据，使用标准 Django 模型（非 UserData）：

| 模型 | App | 说明 |
|------|-----|------|
| `AgentSession` | `agent_service` | Agent 对话会话 |
| `UserPersonalInfo` | `agent_service` | 用户个人信息记忆 |
| `DialogStyle` | `agent_service` | 对话风格配置 |
| `WorkflowRule` | `agent_service` | 工作流规则 |
| `AgentTransaction` | `core` | Agent 操作事务日志（用于回滚） |
| `UserFile` / `UserFolder` / `UserStorageQuota` | `file_service` | 文件存储 |

新增标准 Django 模型时需同步在 `admin.py` 中注册，方便后台调试。
