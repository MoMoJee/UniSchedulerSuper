# 数据模型规范

> 现行版本：2026-07-13。Planner 的唯一事实源是 normalized ORM。`UserData` 中的 `events`、`todos`、`reminders`、`events_groups`、旧 RRule key 和 Planner 操作日志均为 P6 archive，数据库 trigger 禁止写入。

## 1. Planner 主实体

| 模型 | 公开 ID | 要点 |
|---|---|---|
| `EventGroup` | `group_id` | 用户日程组及默认属性 |
| `CalendarEvent` | `event_id` | 单次日程或重复系列 master |
| `Todo` | `todo_id` | 待办；本期不支持 recurrence |
| `Reminder` | `reminder_id` | 单次或重复提醒 master |

它们继承 `PlannerVersionedModel`，具有 `version`、`metadata`、`created_at`、`updated_at`、`deleted_at`。业务 ID 是不可变 UUID 字符串；不得向客户端暴露或依赖 ORM 自增主键。

更新和删除均为乐观锁 + 软删除：命令必须携带 `expected_version`，冲突不覆盖现有对象。

## 2. 时间与时区

- 时间字段以 UTC aware datetime 或 date 存储；转换和序列化统一由 `PlannerTimeCodec` 处理。
- 每个对象保存 `tzid`，默认 `Asia/Shanghai`。API 可以接受带 offset 的 ISO 8601，但客户端不能自行假定数据库为“东八区无时区字符串”。
- 非全天事件使用 `start_at/end_at`，全天事件使用 `start_date/end_date`，二者互斥且受数据库约束。
- 查询窗口是半开区间 `[from, to)`，必须有限，避免无限 RRULE 展开。

## 3. 重复规则

| 类型 | 规则模型 | 稀疏状态模型 |
|---|---|---|
| 日程 | `EventRecurrenceSeries`、`RDate`、`ExDate` | `EventOccurrenceOverride`、`EventRecurrenceSplitReview` |
| 提醒 | `ReminderRecurrenceSeries`、`RDate`、`ExDate` | `ReminderOccurrenceState`、`ReminderDeliveryAttempt` |

- RRULE 保存 canonical 形式；EXDATE/RDATE/override 是独立关系，不拼接到 RRULE 文本。
- 正常 occurrence 由 recurrence engine 在查询窗口内展开，不预生成到 `CalendarEvent` / `Reminder` 表。
- occurrence 标识是服务端产生的完整 `occurrence_ref`（含 `entity_id`、`series_id`、`recurrence_id`、`source_version`）。客户端不得自行拼接。
- “此及以后”分裂通过 lineage 与 review 模型保持可审计；需要 `override_policy` 时由领域命令显式要求，禁止静默丢弃未来 override。

## 4. 关系与协作

- `EventTag`、`TodoTag`、`TodoDependency` 是查询友好关系；依赖环由领域服务拒绝。
- `EventReminderLink`、`TodoReminderLink` 是关联事实源。
- `EventShareGroup` 是 Event 与 `CollaborativeCalendarGroup` 的关系表，不复制事件 JSON；分享成员的 occurrence 查询只读且按 membership 授权。
- `CalendarCollectionVersion` 提供 Feed/CalDAV collection 的版本与 sync token；任何影响该 collection 的写命令必须更新它。

## 5. Cohort、迁移和 archive

| 模型 | 用途 |
|---|---|
| `PlannerMigrationState` / `PlannerMigrationIssue` | 迁移与验证状态 |
| `PlannerCohortAssignment` | 用户/入口 storage mode 与 P6 cutover manifest |
| `PlannerLegacyIdMap` | 旧 ID 到 normalized ID 的追溯映射 |
| `PlannerLegacyWriteGuard` | archive key 的数据库防写规则 |

运行时准入由 `PlannerRolloutPolicy` 依据全局模式、assignment、entrypoint、验证状态及 quarantine 判定。业务代码不得直接把 cohort 状态解释为“可以尝试 legacy”。

## 6. Agent 回滚存储

`PlannerChangeSet` 记录可回滚 Planner 命令的作用范围和前后 hash；`PlannerRollbackSnapshot` 保存当前 `AgentRollbackWindow` 内的压缩 before-snapshot。它们不是永久审计历史：窗口关闭、会话切换、成功回滚或清理任务后可删除。

旧 `AgentTransaction.revision_id` / `django-reversion` Planner 快照不迁移、不用于新回滚。新代码应使用 `change_set_id`、`rollback_window`、`tool_call_id` 与 snapshot checksum。

## 7. `UserData` 的允许范围

`UserData` 仍用于用户设置、界面偏好、Agent 月度 token 摘要等非 Planner 兼容数据。使用时必须遵循 `DATA_SCHEMA` 和默认值约定；请求级 token 明细应写 `AgentUsageRecord`，不得写入大 JSON。

禁止新增 Planner key、直接更新 archive `value`、或以 `MockRequest + UserData.get_or_initialize()` 实现 Planner 读写。
