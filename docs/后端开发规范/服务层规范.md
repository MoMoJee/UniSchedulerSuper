# 服务层规范

> 现行版本：2026-07-13。本文只描述 P6 后的 Planner 服务层。`core/services/`、`MockRequest`、`EventsRRuleManager`、`rrule_engine.py` 和直接操作 Planner `UserData` 的代码属于历史兼容实现，禁止在新功能中使用。

## 1. 唯一调用链

所有 Planner 入口必须遵循同一条链：

```text
HTTP V2 / WebSocket Agent / Quick Action / MCP / 附件 / Feed / CalDAV
  → PlannerExecutionContext
  → PlannerApplicationService
  → PlannerCommandService / PlannerEntityCommandService / QueryService
  → normalized ORM + recurrence engine
```

协议层只负责认证、解析协议字段和映射错误；领域层不接收 Django request、不调用 View、不知道 HTTP/WS/MCP 的具体格式。

## 2. `PlannerExecutionContext`

入口必须自行构造可信上下文，调用方 payload 不得覆盖其中任何字段：

```python
context = PlannerExecutionContext(
    user=request.user,
    source='web_v2',
    entrypoint=PlannerRolloutPolicy.ENTRYPOINT_API_V2,
    request_id=request.headers.get('X-Request-ID', ''),
)
```

`source` 与 `entrypoint` 必须反映真实协议入口。`reversible=True` 只允许 `websocket_agent`，并且必须同时有 `session_id`、`tool_call_id`、`message_index` 与有效 `rollback_window_id`。

## 3. 准入与读写

所有公开 use case 先执行 `PlannerApplicationService.require_access()`：

- 读取只允许 `shadow` 或 `normalized`。
- 写入只允许 `normalized`。
- `blocked` 返回稳定 409；历史测试数据隔离的 `quarantined` 返回 423。
- 不得因 cohort 未准入、缺少迁移数据或异常而 fallback 到 legacy JSON。

## 4. Command 规则

- Event、Todo、Reminder、EventGroup 均是带 `version` 的软删除实体；更新与删除必须传 `expected_version`。
- 读取 occurrence 时必须指定有限 `[from, to)`；正常重复实例是计算投影，不应物化为一行 ORM 数据。
- 重复 Event 的 `single` / `future` 必须使用完整 `occurrence_ref`。`all` 可整体改时间，但不能把系列 DTSTART 改到另一个本地日期；日期或规则从某次开始变化使用 `future`。
- Reminder 的重复实例状态通过 `ReminderOccurrenceState` 表示；目前系列内容更新只支持 `scope=all`，单次状态使用 occurrence action。
- 未知字段、非法 RRULE、非法 scope 和违反领域规则均抛出带稳定 `code` 的 `PlannerCommandError`，由 adapter 映射为 422 或指定 409。

## 5. 事务、版本与回滚

Application Service 的写入口通过 `PlannerSnapshotRecorder.execute()` 管理事务。普通 V2/CalDAV/Quick/MCP 写入只执行领域命令；可回滚的 WebSocket Agent 写入还会：

1. 在同一个数据库事务内保存受影响 aggregate 的压缩 before-snapshot；
2. 创建 `PlannerChangeSet`、`PlannerRollbackSnapshot`、`AgentTransaction`；
3. 回滚时先校验当前 aggregate 的 after hash，冲突则返回 409；
4. 回滚成功后删除该短期 snapshot 并更新日历 collection version。

不要为 Planner 写操作再包 `reversion.create_revision()`，也不要复制整个 `UserData` JSON。`django-reversion` 仅保留给非 Planner 历史对象或后台审计用途。

## 6. Agent / Quick Action / MCP 适配

- Agent 工具通过 `agent_service/tools/planner_application_adapter.py` 构造 `websocket_agent` context。
- Quick Action 使用 `quick_action` context；MCP stdio/HTTP 使用各自 source 与 `mcp_planner` entrypoint。
- 适配层可以把自然语言参数转为领域 payload，但不得重新实现 CRUD、RRule 展开、权限或版本判断。
- 新 Planner 工具不得使用 `@agent_transaction`；该装饰器只保留给非 Planner 的旧兼容事务。

## 7. Feed、CalDAV 与附件

- Feed 调用 `build_calendar_feed()`，只读且使用 URL Token 认证。
- CalDAV 的 PROPFIND/REPORT/GET/PUT/DELETE 仅为协议 adapter；PUT/DELETE 通过 Application Service，必须处理 ETag/If-Match/If-None-Match。
- 内部日程/待办/提醒附件必须通过 Application Service 的 attachment query，用 normalized serializer 生成 Agent 可见内容。
- 不新增对 `core/services/*` 或旧 `views_*` 的 Planner 调用；若需删除历史代码，按 P6 archive 保留期和专项迁移计划执行。
