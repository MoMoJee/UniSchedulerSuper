# 服务层规范

> 本文档描述 UniSchedulerSuper `core/services/` 层的职责和使用约定。

---

## 1. Services 层职责

`core/services/` 提供面向 **AI Agent Tools** 的业务逻辑封装，将复杂的 CRUD 操作从 View 层分离：

| 服务文件 | 封装对象 |
|---------|---------|
| `event_service.py` | `EventService` — 日程创建/更新/删除 |
| `reminder_service.py` | `ReminderService` — 提醒 CRUD |
| `todo_service.py` | `TodoService` — 待办 CRUD |

**使用场景**：Agent Tools（`agent_service/tools/`）调用这些 Service，而不是直接调用 View 函数。  
**View 层**依然直接处理 HTTP 请求，不经过 Service。

---

## 2. MockRequest 模式

Service 层需要模拟 HTTP 请求对象来复用 View 中依赖 `request.user` 的逻辑：

```python
class MockRequest:
    def __init__(self, user):
        self.user = user
        self.is_authenticated = True
```

**规则**：
- `MockRequest` 仅用于 Service 层内部，绝不暴露到 View 或前端。
- `MockRequest` 实例仅需包含 `user` 和 `is_authenticated`，按需扩展。
- 调用 `UserData.get_or_initialize(mock_request, ...)` 时传入 `MockRequest` 实例。

---

## 3. Service 方法规范

所有 Service 方法均为静态方法，统一接受 `user` 对象作为第一参数：

```python
class EventService:
    @staticmethod
    def create_event(user, title, start, end, description="", ...):
        mock_request = MockRequest(user)
        ...

    @staticmethod
    def get_events(user):
        mock_request = MockRequest(user)
        user_events_data, _, _ = UserData.get_or_initialize(
            mock_request, new_key="events", data=[]
        )
        return user_events_data.get_value() or []

    @staticmethod
    def delete_event(user, event_id, delete_scope='single'):
        ...
```

## 4. `reversion` 版本追踪规范

所有修改 `UserData` 的操作（无论在 View 还是 Service 中）均需包裹版本追踪：

```python
import reversion

with reversion.create_revision():
    reversion.set_user(user)
    reversion.set_comment(f"Create event: {title}")
    user_data.set_value(events)
    user_data.save()
```

### 4.1 comment 格式约定

| 操作 | comment 格式 |
|------|-------------|
| 创建事件 | `"Create event: {title}"` |
| 更新事件 | `"Update event: {title}"` |
| 删除事件 | `"Delete event: {event_id}"` |
| 创建待办 | `"Create todo: {title}"` |
| AI Agent 操作 | `"Agent: {operation_description}"` |

### 4.2 `AgentTransaction` 事务记录

Agent Tools 执行的操作额外记录到 `AgentTransaction`，支持 Agent 级别的回滚：

```python
from core.models import AgentTransaction

transaction = AgentTransaction.objects.create(
    user=user,
    session_id=session_id,
    operation_type='create_event',
    operation_data=json.dumps({'event_id': event['id'], ...}),
)
```

回滚接口：`POST /api/agent/rollback/` — 使用 `reversion` 恢复指定时间戳之后的所有修改。

## 5. 分享组数据同步

当事件的 `shared_to_groups` 字段被修改时，必须触发分享组同步：

```python
from core.views_share_groups import sync_group_calendar_data

if shared_to_groups:
    try:
        sync_group_calendar_data(shared_to_groups, user)
    except Exception as e:
        logger.error(f"同步分享组数据失败: {str(e)}")
```

**规则**：分享组同步失败不应影响主操作的成功返回，用 `try/except` 包裹并记录错误日志。

## 6. RRule 引擎使用

重复事件相关操作通过 `EventsRRuleManager`（`core/views_events.py`）或 `RRuleEngine`（`rrule_engine.py`）处理：

```python
from core.views_events import EventsRRuleManager

manager = EventsRRuleManager(user)

# 创建重复事件
main_event = manager.create_recurring_event(event_data, rrule)

# 生成实例（预生成未来 N 天内最多 M 条）
instances = manager.generate_event_instances(main_event, days_ahead=90, max_instances=20)

# 删除单个实例（EXDATE 机制）
events = manager.delete_event_instance(events, event_id, series_id)

# 删除此及以后（UNTIL 截断）
events = manager.delete_event_this_and_after(events, event_id, series_id)
```

### 预生成实例数量约定

| 频率 | days_ahead | max_instances |
|------|-----------|---------------|
| 日 / 工作日 | 90 | 20 |
| 每周 | 180 | 26 |
| 每月 | 365 | 36 |
| 有 COUNT / UNTIL | 365×2 | COUNT 值 / 1000 |

## 7. 文件服务

`file_service/` 提供独立的文件存储服务，核心上传逻辑在 `_core_upload()`：

```python
from file_service.views_api import _core_upload

result = _core_upload(user, file_obj, folder=None, source='upload')
if not result['success']:
    return Response({'error': result['error']}, status=result['status'])
file_instance = result['file']
```

文件名安全处理（`_sanitize_filename`）在上传入口强制执行，防路径穿越。
