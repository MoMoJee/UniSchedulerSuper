## 1. 架构基础

### 1.1 核心机制

OpenCode 采用**请求级上下文组装**模式，每次 LLM 调用都重新构建完整消息序列：

```
[系统提示层] + [对话历史层] + [当前输入层]
     ↓              ↓              ↓
  动态注入      持久化存储      用户当前输入
```

**关键约束**:
- 系统提示层：每次请求重新生成，不保留上一轮配置
- 对话历史层：持久化但可能被压缩/摘要
- 当前输入层：仅包含本次提供的资源

### 1.2 存储架构

```
~/.local/share/opencode/storage/
├── session/{projectID}/{sessionID}.json    # 会话元数据
├── message/{sessionID}/{messageID}.json    # 消息元数据
└── part/{messageID}/{partID}.json          # 内容分片
    ├── TextPart: 文本内容
    ├── ToolPart: 工具调用记录（含输入/输出/状态）
    ├── FilePart: 文件附件引用
    └── ReasoningPart: 推理过程
```

## 2. 上下文分层模型

### 2.1 三层架构详解

| 层级 | 内容 | 生命周期 | 可变性 |
|------|------|---------|--------|
| **L1: 系统提示** | 角色定义、可用技能、工具模式、环境上下文 | 单次请求 | 每轮完全重建 |
| **L2: 对话历史** | 用户消息、Agent回复、工具调用记录 | 跨请求持久化 | 压缩/摘要后保留 |
| **L3: 当前输入** | 用户文本、附件、即时指令 | 当前请求 | 完全由用户控制 |

### 2.2 层级间的数据流动

```
用户操作
    ↓
[L3 当前输入] ←—— 附件、文本、即时指令
    ↓
[上下文组装器] ←—— 读取 L1 配置 + 检索 L2 历史
    ↓
[完整消息序列] → 发送给 LLM
    ↓
LLM 响应
    ↓
[解析器] → 提取工具调用、文本回复、推理
    ↓
[L2 对话历史] ←—— 持久化存储（含工具结果）
```

## 3. 动态组件生命周期

### 3.1 组件状态矩阵

| 组件 | 第一轮存在形式 | 第二轮默认状态 | 跨轮保留条件 |
|------|--------------|--------------|-------------|
| **Skill 定义** | 系统提示中的 `<available_skills>` | ❌ 不存在 | 无（需每轮显式配置） |
| **Tool 定义** | 系统提示中的 function schema | ❌ 不存在 | 无（需每轮显式启用） |
| **Tool 调用记录** | 历史消息中的 `ToolPart` | ⚠️ 可能被压缩 | 作为历史事实保留，但仅作记录 |
| **附件内容** | 当前消息的 `FilePart` | ❌ 不存在 | 无（需每轮重新上传或显式引用） |
| **环境上下文** | 系统提示中的 runtime 描述 | ✅ 重新生成 | 每次请求自动刷新 |

### 3.2 关键结论

**默认情况下，动态组件不会跨轮保留**：

1. **Skills**: 每次请求重新读取 `SKILL.md`，上一轮的技能不会自动延续
2. **Tools**: 每次请求根据当前权限重新组装，禁用后即从上下文中消失
3. **Attachments**: 仅存在于上传时的消息中，后续轮次不会自动包含

## 4. 多轮对话状态管理

### 4.1 状态持久化策略

#### 策略 A: 历史消息隐式保留（有限）

```
[用户] 使用 Skill-A 分析文件 X
[Agent] <调用 Skill-A 获取结果 R>
        ↓
        结果 R 作为 ToolPart 存入历史

[用户] 继续分析（Skill-A 已不在当前系统提示）
[Agent] 只能从历史 ToolPart 中看到"曾调用过 Skill-A"，
        但无法获取 Skill-A 的完整定义和新能力
```

**风险**: 第二轮 Agent 知道"之前用过 Skill-A"，但无法使用 Skill-A 的新功能

#### 策略 B: 显式状态声明（推荐）

```
[系统提示层 - 每轮重建]
当前激活技能: [Skill-A, Skill-B]
当前可用工具: [tool-1, tool-2]
当前关注文件: [file-X, file-Y]  ← 显式声明上一轮的关键附件

[对话历史层]
[之前轮次的摘要...]

[当前输入层]
用户: "继续分析"
```

### 4.2 上下文压缩的影响

当 token 超过阈值时，OpenCode 触发 compaction：

```typescript
// 压缩前
[System Prompt] + [Msg-1] + [Msg-2] + ... + [Msg-N] + [Current]

// 压缩后  
[System Prompt] + [Summary] + [Msg-N-2] + [Msg-N-1] + [Msg-N] + [Current]
                    ↑
              早期消息被摘要化，
              工具调用细节可能丢失
```

**应对方法**: 在 Agent 回复中主动生成"状态快照"

## 5. 显式上下文构建策略

### 5.1 核心原则: 上下文即代码

将上下文构建视为显式编程，而非隐式继承：

```python
# 反例: 假设 Skill 会继承
async def handle_turn(user_input, session):
    # 错误: 假设上一轮的技能仍然可用
    result = await session.prompt(user_input)  # Skill 可能已丢失

# 正例: 显式构建上下文
async def handle_turn(user_input, session, required_skills, active_files):
    # 显式声明当前需要的所有资源
    context = build_context(
        skills=required_skills,      # 明确指定当前技能
        tools=get_enabled_tools(),   # 明确指定可用工具
        file_context=active_files,   # 明确指定关注的文件
        history=session.get_history()
    )
    result = await session.prompt(user_input, context)
```

### 5.2 五类关键场景的构建模式

#### 场景 1: 跨轮 Skill 延续

```
问题: 第一轮用 Skill-A 初始化，第二轮需要继续使用 Skill-A

解决方案:
┌─ 系统提示构建 ──────────────────────┐
│ 角色: 专业分析师                    │
│ 激活技能:                          │
│   - Skill-A (v1.2)                 │ ← 显式声明，而非依赖历史
│   - Skill-B (辅助)                  │
│                                    │
│ 技能状态:                          │
│   Skill-A 上一轮执行结果: [摘要]      │ ← 将上一轮结果作为状态注入
│   当前任务阶段: 分析阶段             │
└────────────────────────────────────┘
```

#### 场景 2: 动态 Tool 切换

```
问题: 第一轮开放 tool-1, tool-2，第二轮关闭 tool-1，但保留 tool-2 的历史结果

解决方案:
┌─ 系统提示构建 ──────────────────────┐
│ 可用工具:                          │
│   - tool-2 (当前启用)               │
│   - tool-3 (新增)                   │
│                                    │
│ 历史工具结果保留:                   │
│   [Tool-1 调用记录 - 仅作参考]       │ ← 历史记录保留，但定义已移除
│   执行时间: 2026-03-13               │
│   关键结果: [摘要]                  │
│   注意: tool-1 当前已禁用，不可调用  │
└────────────────────────────────────┘
```

#### 场景 3: 附件内容延续

```
问题: 第一轮上传 file-X，第二轮需要引用 file-X 的内容但不再上传

解决方案:
┌─ 系统提示构建 ──────────────────────┐
│ 当前关注文件:                       │
│   - file-X (上一轮上传)             │ ← 显式声明文件上下文
│     内容摘要: [第一轮生成的摘要]      │
│     关键片段: [重要行号/段落]        │
│     当前处理位置: 第 50 行            │
│                                    │
│   - file-Y (本轮新增)               │
└────────────────────────────────────┘

注: 若需完整内容，必须重新上传或维护外部文件缓存
```

#### 场景 4: 长会话状态恢复

```
问题: 会话恢复后，如何重建完整的 Agent 状态

解决方案:
┌─ 会话恢复流程 ──────────────────────┐
│ 1. 加载历史消息 (MessageV2)         │
│ 2. 提取最后已知状态:                │
│    - 扫描历史中的 "状态快照" 消息    │
│    - 解析 [AGENT_STATE] 标记         │
│ 3. 重建系统提示:                    │
│    - 根据状态快照激活对应 Skills      │
│    - 恢复关注的文件列表               │
│    - 设置正确的任务阶段               │
│ 4. 生成恢复确认消息                  │
└────────────────────────────────────┘

状态快照格式 (Agent 在回复中生成):
[AGENT_STATE]
phase: "analysis"
active_skills: ["skill-a", "skill-b"]
focus_files: ["main.py", "config.yaml"]
accumulated_findings: ["finding-1", "finding-2"]
[/AGENT_STATE]
```

#### 场景 5: 多 Agent 协作的上下文隔离

```
问题: Agent-A 和 Agent-B 交替执行，如何避免上下文污染

解决方案:
┌─ 上下文隔离策略 ────────────────────┐
│ Agent-A 执行时:                    │
│   系统提示: "你是 Agent-A..."        │
│   激活技能: [Skill-A1, Skill-A2]    │
│   可用工具: [tool-a1, tool-a2]      │
│   历史消息: 仅保留 Agent-A 的消息     │
│                                    │
│ Agent-B 执行时:                    │
│   系统提示: "你是 Agent-B..."        │
│   激活技能: [Skill-B1, Skill-B2]    │
│   可用工具: [tool-b1, tool-b2]      │
│   历史消息: 仅保留 Agent-B 的消息     │
│                                    │
│ 协作接口:                          │
│   通过显式消息传递状态:              │
│   [Agent-A] → 生成结构化输出 → [Agent-B]
└────────────────────────────────────┘
```

## 6. 反模式与陷阱

### 6.1 危险假设清单

| 反模式 | 错误假设 | 实际行为 | 后果 |
|--------|---------|---------|------|
| **Skill 继承** | "上一轮用了 Skill-A，这一轮还在" | 每轮重新注入，默认不包含 | Agent 无法使用 Skill-A 的能力 |
| **Tool 记忆** | "之前禁用的 tool-1，Agent 还记得怎么用" | Tool 定义已从系统提示移除 | Agent 尝试调用不存在的工具 |
| **附件残留** | "上传过的文件，Agent 还记得内容" | 附件仅存在于上传时的消息 | Agent 引用不存在的文件内容 |
| **历史全量** | "对话历史包含所有细节" | 压缩后早期消息被摘要 | 关键细节丢失 |
| **环境恒定** | "工作目录和上一轮一样" | 每次请求重新检测环境 | 外部变更未感知 |

### 6.2 典型错误代码

```python
# ❌ 错误: 假设 Skill 跨轮保留
async def analyze_code(session, code_file):
    # 第一轮: 初始化代码分析技能
    await session.prompt("激活代码分析技能", skills=["code-analyzer"])

    # 第二轮: 假设技能仍然可用
    result = await session.prompt("分析这段代码的复杂度")  # 错误: 技能已丢失
    return result

# ✅ 正确: 每轮显式声明
async def analyze_code(session, code_file):
    # 第一轮
    result1 = await session.prompt(
        "激活代码分析技能",
        context=build_context(skills=["code-analyzer"])
    )

    # 第二轮: 显式保持技能激活
    result2 = await session.prompt(
        "分析这段代码的复杂度",
        context=build_context(
            skills=["code-analyzer"],  # 显式保留
            state={"previous_result": result1.summary()}  # 显式传递状态
        )
    )
    return result2
```

### 6.3 上下文污染案例

```
场景: 用户先要求"用 Python 风格"，然后要求"切换到 JavaScript"

❌ 错误处理:
[系统提示] 你是 Python 专家
[用户] 用 Python 写代码
[Agent] <生成 Python 代码>
[用户] 现在用 JavaScript 写
[Agent] <生成 Python 风格的 JavaScript 代码>  ← 上下文污染！

原因: 历史消息中的 Python 示例影响了当前输出

✅ 正确处理:
[系统提示] 你是 JavaScript 专家  ← 明确覆盖角色
[用户] 现在用 JavaScript 写
[上下文] 忽略之前的 Python 要求，当前任务: JavaScript
[Agent] <生成纯 JavaScript 代码>
```

## 7. 实现模板

### 7.1 上下文构建器模板

```typescript
// context-builder.ts
interface AgentContext {
  role: string;
  activeSkills: Skill[];
  availableTools: Tool[];
  fileContext: FileReference[];
  stateSnapshot: StateSnapshot;
  historyStrategy: 'full' | 'summary' | 'recent';
}

class ContextBuilder {
  private session: Session;
  private config: AgentConfig;

  build(userInput: string, options: BuildOptions): LLMMessage[] {
    return [
      this.buildSystemPrompt(options),  // L1: 显式构建
      ...this.buildHistory(options),     // L2: 策略性选择
      this.buildUserMessage(userInput, options)  // L3: 当前输入
    ];
  }

  private buildSystemPrompt(options: BuildOptions): SystemMessage {
    return {
      role: 'system',
      content: `
# 角色定义
${this.config.roleDescription}

# 当前激活技能 (显式声明)
${options.activeSkills.map(s => `- ${s.name}: ${s.description}`).join('\n')}

# 可用工具集 (当前轮次)
${options.availableTools.map(t => `- ${t.name}: ${t.schema}`).join('\n')}

# 文件上下文 (显式维护)
${options.fileContext.map(f => `
## ${f.name}
- 类型: ${f.type}
- 内容摘要: ${f.summary}
- 处理状态: ${f.status}
`).join('\n')}

# 状态快照 (跨轮传递)
当前阶段: ${options.stateSnapshot.phase}
累积发现: ${options.stateSnapshot.findings.join(', ')}
待处理项: ${options.stateSnapshot.pending.join(', ')}

# 环境上下文 (实时检测)
工作目录: ${this.getWorkingDirectory()}
项目结构: ${this.getProjectOverview()}
      `.trim()
    };
  }

  private buildHistory(options: BuildOptions): Message[] {
    const rawHistory = this.session.getHistory();

    switch (options.historyStrategy) {
      case 'full':
        return rawHistory;  // 仅适用于短会话
      case 'recent':
        return rawHistory.slice(-5);  // 最近 5 轮
      case 'summary':
        return [
          this.createSummaryMessage(rawHistory),
          ...rawHistory.slice(-3)  // 摘要 + 最近 3 轮
        ];
    }
  }

  private buildUserMessage(input: string, options: BuildOptions): UserMessage {
    return {
      role: 'user',
      content: input,
      attachments: options.newAttachments || []
    };
  }
}
```
