> 依据：[核心日程正规化与 RRule 引擎升级方案](./核心日程正规化与RRule引擎升级方案.md)与[当前进展计划](./Planner正规化升级进展与下一阶段计划.md)。

## 1. 不可突破的实施边界

1. legacy JSON 在 P3 cohort 切换完成前仍是旧入口事实源；不得静默混写两套存储。
2. v2 GET 仅允许 effective mode 为 `shadow` 或 `normalized`；v2 command 仅允许 `normalized`。
3. 正常 recurrence occurrence 永不写为 `CalendarEvent`/`Reminder` 实例行。
4. MoMoJee 原始 legacy 行、checksum 和 issue 证据必须保留；修复优先改迁移解释或显式 mapping，不直接覆盖源 JSON。
5. User1、test_user、User15、User21、User22 保持 quarantine；只记录 `retired-test-data` 处置，不标 verified/resolved。
6. Agent、Quick Action、MCP、附件解析与精确回滚属于 P4；Feed/CalDAV 属于 P5。本阶段只保证它们的 legacy compatibility 不被破坏。

## 3. P3-0：v2 cohort 安全门禁

### 3.1 实施步骤

1. 在 `PlannerRolloutPolicy` 中定义稳定入口名 `api_v2`、`web_calendar`、`web_todo`、`web_reminder`、`web_search`、`web_share`、`course_import`。
2. 提供统一 access decision：
   - `legacy`：v2 读写均拒绝；
   - `shadow`：只读允许、写入拒绝；
   - `normalized`：读写允许；
   - 未 assignment/未登记 entrypoint/未 verified/存在 issue：全部拒绝。
3. 所有 v2 view 使用同一 guard；错误返回稳定 `code`，不得自行只调用 `is_verified_clean()`。
4. 拒绝请求不得创建 event、override、ChangeSet、CalendarChange 或 collection version。

### 3.2 测试范围

- 全局 legacy/shadow/normalized × assignment 缺失/entrypoint 缺失/legacy/shadow/normalized × clean/issue。
- GET definitions/occurrences/search 与 POST/PATCH/DELETE。
- shadow GET 成功且零写；shadow command 返回 409/403 且零写；normalized command 成功。

### 3.3 验收标准

- 未显式登记的生产用户无法访问任何 v2 endpoint。
- `PlannerRolloutPolicy` 是唯一准入判定，view 不重复推断模式。
- 门禁矩阵和全量旧测试通过。

## 4. P1-C：基础契约与压力测试补强

### 4.1 实施步骤

1. 补 codec/expander fixture：MONTHLY/YEARLY、WKST、负 BYMONTHDAY、序号 BYDAY、BYSETPOS、BYWEEKNO、BYYEARDAY、多 RDATE/EXDATE。
2. 校验 DATE 与 DATE-TIME 类型一致，非法混用显式失败。
3. 增加 10,000 个潜在 slot 的有限窗口测试，记录运行上限并验证无持久化写入。
4. 增加模型/command 验证：跨 user group/link/dependency、Todo dependency 环、override 时间形状、version conflict、soft delete、collection token 单调性。
5. 重跑 direct UserData access 报告。

### 4.2 测试范围

- 纯 unittest：codec、expander，不依赖数据库/当前时间。
- Django TestCase：repository 零写、关系所有权、command 原子性。
- 管理命令：`report_planner_direct_userdata_access`。

### 4.3 验收标准

- 受支持 RRULE 子集每类至少一个正例和一个非法边界例。
- 无限规则的窄窗口展开数量只与窗口相关；模型计数前后不变。
- `planner_bypass_count = 0`。

## 6. P3-C1：Event recurrence 命令完成

### 6.1 实施步骤

1. 实现 `this_and_future` split：验证 recurrence slot，计算父规则 `< recurrence_id` 的 COUNT/UNTIL 截断，创建 child master/series/lineage。
2. 强制未来 override policy：
   - `keep_as_single`：转为独立 event；
   - `discard_with_audit`：删除但写完整 ChangeSet；
   - `map_by_ordinal`：只有引擎证明一一对应时允许。
3. 实现 recurrence detach/cancel：single occurrence 转独立 event、all 取消 recurrence、future 从锚点转单次/新系列。
4. 所有操作在同一事务内更新 event/series/RDATE/EXDATE/override/collection/change/audit。

### 6.2 测试范围

- DAILY/WEEKLY/MONTHLY、COUNT、UNTIL、无限规则。
- split 边界当天、移动 override、取消 override、未来 override 三策略。
- stale version、非法 recurrence id、跨用户 ref、事务中途异常回滚。

### 6.3 验收标准

- `single/all/this_and_future` 创建/改/删全部有原子测试。
- split 前后 occurrence 集符合选定策略，父 UID 稳定，child lineage 完整。

## 8. P3-C3：搜索、冲突、共享与课程导入

## 9. P3-D：Web 适配与 cohort 切换

### 9.1 实施步骤

1. 新增统一前端 `planner-v2-client` 和 occurrence adapter；所有请求带 CSRF、错误码与 expected version。
2. FullCalendar source 改为 v2 window API；event id 仅作渲染 key，完整 ref 放 `extendedProps`。
3. event click/drop/resize/create/edit/delete 使用 scope command；失败时 revert 并刷新窗口。
4. Todo、Reminder、Group、Share、Search、Import manager 按 rollout bootstrap 决定 legacy/v2；同一页面会话固定模式，禁止逐请求摇摆。
5. 后端 home/bootstrap 返回当前用户各入口 effective mode；旧用户保持 legacy。
6. 在隔离数据库执行浏览器 smoke/E2E，再登记 shadow read；read diff 零后登记 normalized write。

### 9.2 浏览器测试范围

- 月/周/日视图窗口加载、切换范围、刷新零写。
- event 单次/全系列/此及以后拖拽、resize、编辑、删除与冲突提示。
- Todo CRUD/转换、Reminder action、搜索跳转、共享只读、课程导入。
- 版本冲突刷新、403/409 错误提示、cohort 回退后 legacy 页面可用。

### 9.3 验收标准

- normalized cohort 的日历、Todo、Reminder、Group、Search、Share、Import 全部走 v2。
- legacy cohort 不受影响；任何入口不得混读/混写。
- 多次窗口读取不增加 event/reminder/state/override 行数。

## 14. 生产 cohort 补充收口（2026-07-12）

手工验收发现 MoMoJee 尚无 cohort assignment，因此页面仍走 legacy：创建无 COUNT/UNTIL 的 DAILY series 时，旧服务把未来 90 天预生成成 91 条 `UserData.events`，UI 到 2026-10-11 后不再显示。用户删除该测试 series 后，91 条实例和 registry 项均消失，`events`/`events_rrule_series` checksum 精确恢复，未留下 normalized 行、ID map、ChangeSet 或 unresolved issue。

根因和修复：

1. 原 E2E 只给临时账号配置 normalized cohort，验证了 v2 无限窗口引擎，却没有验证正式 MoMoJee 是否完成 cohort 切换。
2. rollout 原先只看 migration state 状态，不实时比对源 row id/checksum。现已在每次准入时核对所有 legacy Planner 源；verified 后源漂移会立即回退 legacy。
3. 全局默认改为 `normalized`；未 assignment 用户仍由 policy 回退 legacy。MoMoJee 在 strict/parity 再次 0 diff 后登记七个 normalized entrypoint。
4. normalized assignment 增加后端 legacy Planner 写硬阻断，防止漏网前端路径静默双写。
5. Web adapter 改为按 `web_calendar/web_todo/web_reminder/web_search/web_share/course_import` 分别判断，不再用笼统 `api_v2` 模式。
6. 补齐 Todo 象限拖拽、提醒状态/延后/通知/日历操作、课程导入内建组的 normalized 分支；暂未开放的重复提醒单次/未来编辑会明确阻止，不会回落 legacy。
